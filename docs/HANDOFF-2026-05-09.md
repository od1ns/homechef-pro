## Qué es HomeChef Pro

SaaS para cocineros caseros venezolanos: el chef gestiona menú, recetas
y costos desde una web (admin_web), el cliente final pide platos por la app
Android (client_app), y la cocina ve órdenes en una tablet (kitchen_tablet).
Pagos manuales con comprobante (Pago Móvil VES, Zelle, transferencia),
facturación SENIAT/IGTF (provider mock en dev).

## Stack

- **Backend** .NET 10, Clean Architecture (Domain / Application / Infrastructure / Api).
- **DB** PostgreSQL 16, schema en SQL puro (sin EF Migrations). snake_case en todo.
- **Cache** Redis 7.
- **Frontend** Flutter — tres apps que comparten el paquete `shared/`:
  - `admin_web` (chef en navegador, puerto 8090)
  - `client_app` (cliente móvil Android + web demo, 8091)
  - `kitchen_tablet` (cocina, 8092)
- **Auth** ASP.NET Identity + JWT Bearer + refresh tokens + TOTP 2FA opcional.
- **Deploy** Docker Compose (postgres + redis + api + nginx + certbot).

## Gotchas conocidos (acumulados)

1. **Contraseña admin real**: `w+OC-9uhlJ0HRZ=bzcLsap5F` (del `.env.staging.local`). El bootstrap `demo1234` no matchea el hash en staging.
2. **Quick tunnel cambia cada restart**: actualizar ALLOWED_HOSTS y CORS en `.env.staging.local` y reiniciar API.
3. **`http.sys` de Windows rechaza hosts desconocidos**: cloudflared necesita `--http-host-header localhost`.
4. **`DbUpdateConcurrencyException` en commands que modifican aggregates con triggers**: cargar con `AsNoTracking()` y agregar child entities directamente al DbSet. Aplica a cualquier aggregate que tenga `trg_*_touch` en SQL.
5. **`internal` vs `public` en Domain**: métodos llamados desde Application deben ser `public`. Los que solo los llama el mismo aggregate pueden quedar `internal`.
6. **`if ... else for` en Dart collection literals**: no válido. Usar dos `if` separados con spread `...[...]`.
7. **`git push` y locks**: hacerlo desde PowerShell del host, no desde el sandbox Linux.
