## Qué es HomeChef Pro

SaaS para cocineros caseros venezolanos: el chef gestiona menú, recetas
y costos desde una web (admin_web), el cliente final pide platos por la app
Android (client_app), y la cocina ve órdenes en una tablet (kitchen_tablet).
Pagos manuales con comprobante (Pago Móvil VES, Zelle, transferencia),
facturación SENIAT/IGTF (provider mock en dev).

## Stack

- **Backend** .NET 10, Clean Architecture (Domain / Application / Infrastructure / Api).
- **DB** PostgreSQL 16, schema en SQL puro (sin EF Migrations). snake_case en todo.
- **Cache** Redis 7.
- **Frontend** Flutter — tres apps que comparten el paquete `shared/`:
  - `admin_web` (chef en navegador, puerto 8090)
  - `client_app` (cliente móvil Android + web demo, 8091)
  - `kitchen_tablet` (cocina, 8092)
- **Auth** ASP.NET Identity + JWT Bearer + refresh tokens + TOTP 2FA opcional.
- **Deploy** Docker Compose (postgres + redis + api + nginx + certbot).

## Patrones de código importantes

- **CQRS con MediatR** en Application. Cada caso de uso = `Command` o `Query` + `Handler`. Validación con FluentValidation pipeline behavior.
- **Endpoints minimal** en `Api/Endpoints/*Endpoints.cs`, agrupados por bounded context (`AdminRecipesEndpoints`, `ClientMenuEndpoints`, etc.).
- **Authorization policies**: `Admin`, `Cashier` (incluye Admin), `Cook`, `Client`. Endpoints públicos usan `AllowAnonymous` + rate limiting (group "public").
- **Uploads** F-09: magic byte check + filename regex strict + canonical path verification para defense-in-depth contra path traversal. Storage local en `LocalFileStorage` con prefijo `chef_id`.
- **JWT options** se validan en startup con `IValidateOptions<JwtOptions>` que detecta placeholders, claves cortas y vacías. Falla rápido si no hay `JWT_SIGNING_KEY` real.
- **Theme Flutter** — `HcpThemeExtension` con `palette.{ink,inkSoft,inkMuted,bg,card,line,red,redSoft,accent}`. Importa con `import 'package:homechef_shared/homechef_shared.dart'`.
- **API client Flutter** — `HcpApi` envuelve `ApiClient`. Errores tipados como `ApiException`. Refresh token automático en 401.

## Gotchas conocidos

1. **Quick tunnel cambia cada restart**: Cloudflared sin cuenta genera URL nueva cada vez. Hay que actualizar `.env.staging.local` y rebuildar AAB cada vez. Solución definitiva: dominio propio + tunnel nombrado (Sesión B pendiente).
2. **http.sys de Windows rechaza hosts desconocidos**: Cloudflared a docker en Windows necesita `--http-host-header localhost`. Sin eso falla con HTTP 400 IIS.
3. **wslrelay duplicado en `[::1]:8080`**: hay dos LISTENERS en 8080, uno en `0.0.0.0` (docker backend) y otro en `[::1]` (wslrelay). Si `--url http://localhost:8080` resuelve a IPv6, va al wslrelay. Usar `127.0.0.1` explícito.
4. **Dart `String.fromEnvironment`**: el define se llama `HCP_API_BASE`, no `API_BASE_URL`. Si pasás el nombre incorrecto cae al default `http://localhost:8080` y la app no conecta en celular.
5. **flutter analyze vs build real diverge**: el analyzer dice "unnecessary_import: package:flutter/foundation.dart" en `client_app/lib/utils/receipt_share.dart` pero al compilar real falla con "Undefined name kIsWeb / Uint8List". El import lleva `// ignore: unnecessary_import` para mantenerse.
6. **Archivos truncados en disco**: en este chat hubo dos archivos cortados al guardar — `client_app/lib/screens/menu_screen.dart` (cortado en `Icons.restau`) y `admin_web/lib/screens/recipe_editor_screen.dart` (cortado en `BorderRad`). Ambos arreglados, pero conviene verificar `git diff` antes de commit.
7. **CRLF/LF**: se normalizó con `.gitattributes` en Fase 0; archivos Dart son LF.
