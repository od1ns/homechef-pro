Info
Resumen ejecutivo. La fundación de auth está bien diseñada (JWT HMAC-SHA256, refresh tokens 64 bytes random hasheados, validación estricta de issuer/audience/lifetime, todos los endpoints admin protegidos). Sin embargo, hay 3 hallazgos críticos que bloquean el go-live: F-01 docker-compose fuerza Development en cualquier entorno; F-02 los archivos de /uploads/ se sirven sin auth (modo #6 del premortem); F-03 la JWT signing key tiene un placeholder literal en appsettings.json committed. Adicionalmente, faltan capas estándar: rate limiting, security headers, magic-byte validation en uploads y MFA para Admin/Cashier. Auditoría apunta a ASVS 5.0 Level 2.

Threat model

Atacantes

Anónimo, cliente registrado, admin/cashier comprometido, insider con BD, MITM en wifi, supply-chain.

Trust boundaries

Flutter ↔ API /api/*; navegador ↔ /uploads/* static; webhook delivery ↔ API; CI/CD ↔ deploy host; BD ↔ API.

Sensitive assets

JWT key, refresh tokens, comprobantes (datos bancarios), creds admin, RIF/cédula, direcciones, password hashes, connection strings, webhook secrets.

Top MITRE ATT&CK

T1078 Valid Accounts (bootstrap demo1234), T1190 Exploit Public-Facing App (uploads leak), T1110 Brute Force (sin rate limit).

Findings

F-01

docker-compose.yml fuerza ASPNETCORE_ENVIRONMENT: Development

CRITICAL Effort: small A02:2025 Misconfig CWE-489
Componentedeploy/docker-compose.yml
Atacanteanónimo de internet
El compose fuerza Development sin override. Esto activa MapOpenApi() y MapScalarApiReference() (docs interactivas públicas en /scalar/v1), CORS que acepta cualquier localhost con AllowCredentials, y verbosidad de errores. Si el compose se reusa en staging/prod, todo eso queda expuesto.
Exploitation
  1. Atacante visita /scalar/v1: obtiene mapa completo de la API.
  2. Identifica todos los endpoints AllowAnonymous (login, refresh, register, payment-proofs upload).
  3. Excepciones no manejadas filtran detalles.
Fix
  • En docker-compose: ASPNETCORE_ENVIRONMENT: ${ASPNETCORE_ENVIRONMENT:-Production}.
  • Crear docker-compose.staging.yml separado.
  • Confirmar que MapOpenApi/Scalar solo cargan en IsDevelopment().
Verification
curl https://staging.homechef.app/scalar/v1 → 404. docker compose config | grep ASPNETCORE_ENVIRONMENT → Production o Staging.
F-02

Static files de /uploads/* servidos sin autenticación

CRITICAL Effort: medium A01:2025 BAC API1:2023 BOLA CWE-284
ComponenteProgram.cs líneas 65-72
Atacanteanónimo, cliente registrado
app.UseStaticFiles() con PhysicalFileProvider sirve uploads/ directamente, sin pasar por la pipeline de auth. Solo mitigado por filename Guid.NewGuid("N") (128 bits, no enumerable). Pero cualquier filtración del GUID (logs, respuestas, screenshots, BD) expone el archivo. Es el modo #6 del premortem.
Exploitation
  1. Cliente A tiene URL /uploads/payment-proofs/abc123def....
  2. URL filtrada en log centralizado o screenshot.
  3. Cualquiera con la URL accede al comprobante (datos bancarios visibles).
  4. Post multi-tenant: IDOR cross-tenant directo.
Fix
  • Quitar app.UseStaticFiles({ RequestPath: "/uploads" }).
  • Endpoint autenticado GET /api/uploads/payment-proofs/{filename} con verificación de owner + tenant.
  • Alternativa: nginx auth_request hacia la API antes de servir.
  • Considerar signed URLs con expiración de 5 min.
Verification
Test Should_Reject_Anonymous_PaymentProofAccess → 401. Test Should_Reject_CrossUser_PaymentProofAccess → 403.
F-03

JWT:SigningKey literal en appsettings.json committed

CRITICAL Effort: small A04:2025 Crypto CWE-321
Componenteappsettings.json línea 19
Atacantecualquier lector del repo
El default es "REEMPLAZAR_EN_PRODUCCION_CON_SECRETO_LARGO_Y_ALEATORIO_DE_AL_MENOS_32_BYTES". La validación length < 32 pasa (75 chars). Si en prod no se sobreescribe vía env var, los JWT se firman con secreto público → cualquiera forja tokens admin.
Fix
  • Vaciar el campo: "SigningKey": "".
  • Validación adicional: rechazar valores que empiecen con "REEMPLAZAR", "change-me".
  • Leer key SOLO desde env var JWT_SIGNING_KEY en non-Development.
  • CI check: si appsettings.json contiene "REEMPLAZAR" → falla.
  • Rotar la key actual en deploy/.env.
Verification
Test que arranca app con Jwt:SigningKey="REEMPLAZAR_..." → debe fallar startup.
F-04

Rate limiting completamente ausente

HIGHEffort: small A07:2025API4:2023CWE-307
grep "AddRateLimiter\|UseRateLimiter" → 0 resultados. Login, register, refresh y uploads son brute-forceables/abuse-ables. Combinado con F-07 (bootstrap admin demo1234), credential stuffing rompe la cuenta admin en segundos.
Fix
builder.Services.AddRateLimiter(o => {
  o.AddFixedWindowLimiter("auth", x => { x.PermitLimit=10; x.Window=TimeSpan.FromMinutes(1); });
  o.AddFixedWindowLimiter("upload", x => { x.PermitLimit=20; x.Window=TimeSpan.FromMinutes(1); });
  o.RejectionStatusCode = 429;
});
app.UseRateLimiter();
// .RequireRateLimiting("auth") en login/register/refresh
Verification
Test integration con 12 logins rápidos → 11vo y 12vo devuelven 429.
F-05

Security headers ausentes (HSTS, CSP, X-Frame-Options, X-Content-Type-Options)

HIGHEffort: small A02:2025CWE-693
No hay UseHsts(), UseHttpsRedirection(), ni middleware de security headers. Si nginx no compensa: MITM via downgrade, click-jacking, XSS via uploads con MIME sniffing.
Fix
Middleware con X-Content-Type-Options: nosniff, X-Frame-Options: DENY, Referrer-Policy: no-referrer, Permissions-Policy minimal. UseHsts + UseHttpsRedirection en non-dev. Considerar paquete NetEscapades.AspNetCore.SecurityHeaders para CSP.
F-06

Connection string con password en appsettings.json committed

HIGHEffort: small A04:2025CWE-798
"PostgreSQL": "...Password=homechef_dev" committed. Aunque sea valor de dev, normaliza la práctica peligrosa y queda como blast radius si LFI/SSRF lee el JSON.
Fix
Vaciar campo. Inyectar siempre vía env var ConnectionStrings__PostgreSQL. User Secrets en dev. Validación al startup que rechaza homechef_dev en non-Development.
F-07

Bootstrap admin con password "demo1234" en deploy/.env

HIGHEffort: medium A07:2025CWE-1392
Si jacques o el chef olvidan cambiar la password antes de exponer a internet, hay ventana donde admin@homechef.local / demo1234 es válido. Sin rate limit (F-04), credential stuffing rompe esto en segundos.
Fix
Generar password random al bootstrap, escribir a archivo 0600. Forzar MustChangePassword=true. O policy de password que rechace literal demo1234.
F-08

Webhook secrets vacíos por defecto sin enforcement

HIGHEffort: small A08:2025API10:2023CWE-345
DELIVERY_SECRET_*= vacíos en .env. Si llegan vacíos a prod con RejectInvalidSignature=true, depende de la implementación del verifier qué pasa. Pendiente: leer DeliveryWebhookSignatureVerifier.cs.
Fix
Validación al startup: si RejectInvalidSignature=true Y algún secret está vacío → throw. O endpoint responde 503 "not configured" si secret vacío.
F-09

Upload de comprobantes: solo Content-Type, sin magic bytes ni re-encode

HIGHEffort: medium A02/A05:2025CWE-434
Solo se valida file.ContentType del request (spoofeable). Sin magic bytes, sin re-encode con ImageSharp, sin validación de SVG. Combinado con F-02 → vector de XSS stored. La extensión se deriva del Content-Type del request, no del contenido real.
Fix
Validar magic bytes (primeros 8-12 bytes). Re-encode con SixLabors.ImageSharp: si decode falla, rechazar. Derivar extensión del formato detectado. Servir con Content-Type real + nosniff. Considerar dominio cookieless (cdn.homechef.app).
F-10

POST /api/uploads/payment-proofs es AllowAnonymous (DoS y abuse)

HIGHEffort: medium A06:2025API4:2023
Anonymous por diseño (guest customer). Combinado con F-04: storage exhaustion DoS, archivos huérfanos sin cleanup, costo si va a S3 en el futuro.
Fix
Rate limit 10/min/IP, 50/h/IP. Cron de cleanup de uploads no asociados a order_id. Reducir limit de 5 MB a 1.5 MB. Considerar Turnstile/hCaptcha para anonymous.
F-11

AccessTokenMinutes = 60 es ventana grande

MEDIUMEffort: trivial
1 hora de validez de access token. Token comprometido (logs, XSS) tiene ventana de uso de 60 min. Estándar moderno: 15 min con refresh rotation (que ya tienen).
Fix
Bajar AccessTokenMinutes a 15.
F-12

/health/db filtra counts del negocio (anonymous)

MEDIUMEffort: trivialA09:2025
Endpoint anónimo retorna ingredients y recipes count. Filtra estado del negocio a competidores.
Fix
Solo { status: "ok", db: "postgresql" }. Mover counts a /api/admin/dashboard/health.
F-13

AllowedHosts: "*" permite host header injection

MEDIUMEffort: trivial
Cualquier Host header aceptado. Riesgo de password reset poisoning (si se agrega esa feature) y cache poisoning.
Fix
En prod: "AllowedHosts": "homechef.app;www.homechef.app". En dev: "*".
F-14

CORS AllowCredentials con fallback a localhost en non-dev

MEDIUMEffort: small
Si Cors.AllowedOrigins está vacío Y IsDevelopment(), acepta cualquier localhost. Si por error ASPNETCORE_ENVIRONMENT=Development queda en prod (ver F-01), CORS abierto a localhost del atacante.
Fix
Validar al startup en non-dev que Cors.AllowedOrigins no esté vacío. Loguear warning si lo está.
F-15

Posible account enumeration por timing en /api/auth/login

MEDIUMEffort: smallCWE-208
Mensaje uniforme ("Invalid email or password"), bien. Pero diferencia de timing observable: user no existe = O(1), user existe + password mal = O(bcrypt) ~100ms.
Fix
Dummy hash compute en path "user not found": _hasher.HashPassword(null!, "dummy") para igualar tiempos. Test de regresión: diff < 20ms con n=50.
F-16

POST /api/auth/logout es AllowAnonymous

MEDIUMEffort: small
Defendible (refresh token plain en body autentica), pero sin user_id en logs. Vector de DoS con dictionary de refresh tokens leaked.
Fix
RequireAuthorization(). Validar que el refresh token enviado pertenece al user del access token.
F-17

No MFA en cuentas Admin/Cashier

MEDIUMEffort: largeASVS V2.7
Admin del chef es el único aprobador de pagos y cambios de stock. Password comprometida (phishing/leak/brute-force) = control total.
Fix
TOTP (Authenticator app) para Admin y Cashier antes del piloto. Identity tiene soporte nativo.
F-18

Sin lockfile de NuGet (packages.lock.json)

LOWEffort: smallA03:2025
Sin lockfile, builds reproducibles no garantizadas. Version-range en transitivas habilita supply chain attacks.
Fix
<RestorePackagesWithLockFile>true</> en csproj, dotnet restore --use-lock-file. CI con --locked-mode.
F-19

Sin scanning de dependencias en CI

LOWEffort: small
No Dependabot, no Snyk, no dotnet list package --vulnerable.
Fix
Agregar dependency-review-action en PRs. Workflow nightly con dotnet list package --vulnerable.
F-20

Sin SBOM ni provenance (post-piloto)

INFOEffort: mediumSLSA L2
Pre-deploy a piloto: bueno-tener. Pre-SaaS a 3-10 chefs: esperable.
Fix
microsoft/sbom-tool en build. Cosign sign de imágenes Docker. SLSA L2 generator.

Attack chains

CH-1 — RCE via misconfigured Development env + JWT placeholder

Goal: ejecución remota de código en el VPS del chef piloto.

Pre-conditions: F-01 + F-03.

  1. Atacante visita /scalar/v1 (activo por F-01) → mapa completo de la API.
  2. Lee el repo, encuentra el placeholder en appsettings.json.
  3. Forja JWT con esa key, role Admin.
  4. Si en prod no se sobreescribió Jwt__SigningKey, token aceptado.
  5. Admin total → potencial escalada a RCE via deserialization en uploads (a verificar en Pasada B).

MITRE ATT&CK: T1190 → T1078 → potencial T1059

CH-2 — Comprobantes filtrados via /uploads + log central

Goal: robar comprobantes de pago.

Pre-conditions: F-02 + filtración de GUID.

  1. Atacante consigue al menos un GUID (logs, screenshot, etc.).
  2. GET /uploads/payment-proofs/<guid> → 200 con la imagen.
  3. Si timestamps prefix predecible, itera GUIDs cercanos.
  4. Acumula 100+ comprobantes con datos bancarios.
  5. Thread en X. Reputación destruida.

MITRE ATT&CK: T1213 (Data from Information Repositories)

CH-3 — Account takeover de admin via brute force + bootstrap default

Goal: controlar la cuenta admin.

Pre-conditions: F-04 + F-07 + chef no cambió password.

  1. Tras deploy, atacante POST /api/auth/login con dictionary.
  2. demo1234 está en top-100. Acierta en < 100 intentos.
  3. Sin lockout ni rate limit, no hay alerta.
  4. Admin total. Aprueba pagos falsos, descarga comprobantes (F-02).

MITRE ATT&CK: T1110.001 → T1078
