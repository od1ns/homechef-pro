## 2. Threat model — perimetro post-deploy

**Atacantes considerados al pasar a internet:**

- **Bot scanner:** internet-wide GETs / POSTs a endpoints conocidos (/admin, /actuator, /env, /.git). Mitigaciones: nginx 404 por default, ASP.NET routing tampoco matchea esos paths, rate limit global 120/min/IP.
- **Cliente legitimo abusivo:** crear orders masivas, subir comprobantes basura, brute-force de tokens de orders (F-24). Mitigaciones: rate limits per-IP, F-31 limites en cantidad/items, F-09 magic bytes.
- **Atacante con WhatsApp del chef:** captura URLs de orders, reenvia a quien no debe. Mitigacion: F-24 access_token de 64 chars firmado por SQL trigger.
- **Operador del SaaS comprometido:** persona con acceso a env vars puede leer todo. Mitigacion: secretos rotables, MFA admin (F-17 — pendiente, riesgo aceptado).
- **Proveedor de delivery comprometido:** webhook spoof. Mitigacion: F-32 HMAC firma con secret per-provider, RejectInvalidSignature=true en prod.

**Trust boundaries verificados:**

- Internet ↔ nginx (rate limiting 10/s, body size 10MB, ACME challenge separado).
- Nginx ↔ API container (HTTP en red interna docker, no expuesto a host en prod).
- API ↔ Postgres (red interna docker, password env var).
- API ↔ Redis (red interna docker, password env var).
- Cliente Flutter ↔ API: HTTPS-only en prod, JWT bearer + refresh rotation, CORS de dominios fijos.

## 4. Hallazgos residuales

### D-01 [Medium / Bloqueante a deploy] `Cors:AllowedOrigins` vacio en appsettings.json

**Descripcion.** En produccion, si `Cors:AllowedOrigins` queda vacio y el environment es `Production`, el codigo en `Program.cs:115-137` no entra al branch de dev (que permite localhost wildcard) y termina con un policy sin origins configurados. Esto se comporta como "deny all cross-origin" — fail-closed, lo cual es correcto desde la perspectiva de seguridad pero rompe la funcionalidad: las apps Flutter del cliente no podran consumir la API.

**Mitigacion.** Operador debe definir `Cors__AllowedOrigins__0=https://app.homechef.app`, `Cors__AllowedOrigins__1=https://admin.homechef.app`, etc. via env var antes del primer arranque.

**Costo.** 5 minutos en runbook. **Estado: bloqueante de deploy, no de codigo.**

### D-02 [Medium / Bloqueante a deploy] `AllowedHosts` debe definirse por env var

**Descripcion.** F-13 (Pasada A) hace que `Program.cs:43-50` haga throw si `AllowedHosts == ""` o `"*"` en non-dev. Si el operador olvida definirla, el container falla en startup — buena defensa, pero requiere accion explicita.

**Mitigacion.** `AllowedHosts=homechef.app;www.homechef.app;api.homechef.app` en `deploy/.env`. Mencionar prominente en runbook.

### D-03 [Medium / Bloqueante a deploy] HTTPS cert + nginx config final

**Descripcion.** `deploy/nginx/conf.d/api.conf` tiene la seccion HTTPS comentada (template). Operador debe correr `scripts/init-letsencrypt.sh` para emitir certs Let's Encrypt y luego activar el redirect HTTP→HTTPS. Mientras los certs no esten, nginx sirve HTTP plano (vulnerable a MITM en captura de credenciales).

**Mitigacion.** El runbook debe incluir 3 pasos:
1. `docker-compose -f docker-compose.yml -f docker-compose.prod.yml up -d` (servicios sin HTTPS).
2. `bash scripts/init-letsencrypt.sh homechef.app` (emite certs por dominio).
3. Edit `nginx/conf.d/api.conf` activando block 443 y descomentar `return 301 https://...` en block 80.

**Riesgo si se omite.** Captura de JWT en transit en redes hostiles. Para el piloto inicial (chef especifico, dispositivos suyos) es bajo, pero debe cerrarse antes de marketing publico.

### D-04 [Medium] Bootstrap admin password debe ser fuerte

**Descripcion.** F-07 rechaza `demo1234/admin/password` literal. No rechaza otros passwords debiles (8 chars con todos numeros, palabra del diccionario, etc.). El operador podria poner `Caracas2026` y pasaria el check.

**Mitigacion.** Generar con `openssl rand -base64 24` y guardar en password manager. Comentario en `deploy/.env.example` (si existe) sugiriendo el comando.

**Recomendacion adicional.** Forzar cambio de password en primer login del admin (no implementado hoy — riesgo aceptado).

### D-05 [Medium] Webhook secrets vacios bloquean acceptance hasta configurar

**Descripcion.** En `appsettings.json`, los 3 secrets de delivery webhooks (yummy/pedidosya/rappi) estan vacios. F-32 con `RejectInvalidSignature=true` (default en prod) hara que cualquier webhook entrante reciba 401 hasta que el operador configure los secretos compartidos con cada proveedor.

**Mitigacion.** Si el chef piloto no usa delivery (solo retiro en local), no es relevante. Si usa, coordinar el secret con cada proveedor antes de habilitar el feature en la app.

**Estado.** No bloqueante si se acepta retiro-only para el piloto.

### D-06 [Low] CSP (Content-Security-Policy) no implementado

**Descripcion.** El sistema sirve solo API JSON (no HTML), asi que CSP no aporta proteccion contra XSS clasico. Sin embargo, OpenAPI/Scalar UI estan expuestas en dev y *podrian* exponerse accidentalmente en prod si alguien commitea el `if (app.Environment.IsDevelopment())` mal.

**Mitigacion.** Trivial: agregar `Content-Security-Policy: default-src 'none'` al middleware de headers cuando la respuesta no sea HTML. Costo: 10 lineas.

**Estado.** Aceptable para piloto.

### D-07 [Low] Logs solo en console (no archivo / no agregador)

**Descripcion.** Serilog escribe a stdout. El container Docker captura stdout en el daemon, pero no hay rotacion ni envio a Loki/CloudWatch/etc. Si el chef piloto necesita audit trail (SENIAT exige preservar registros), depender de docker logs puede no escalar.

**Mitigacion.** Para el piloto: `docker compose logs > /var/log/homechef-api-$(date +%F).log` cron job (1 linea de bash). Para escalar: agregar Serilog.Sinks.File que ya esta en csproj pero no configurado.

**Estado.** Aceptable. Documentar el cron en runbook.

### D-08 [Low] No hay test de "startup failure" en non-dev

**Descripcion.** F-06/F-07/F-13 son validaciones de Program.cs que disparan al arrancar. No hay test que verifique que el container falla al arrancar con configuracion mala. Si alguien remueve esos throws, las regression tests pasan igual.

**Mitigacion.** Agregar `Startup_should_throw_if_pg_password_is_dev_in_non_development` test que crea un WebApplicationFactory con `UseEnvironment("Production")` + dev connection string, asserta que CreateClient() lanza InvalidOperationException. Costo: 1 hora.

**Estado.** Aceptable. Las validaciones existen y se han revisado manualmente.

### D-09 [Info] Admin endpoints sin rate limit explicito

**Descripcion.** Los endpoint groups `/api/admin/*` no tienen `RequireRateLimiting(...)` y caen al GlobalLimiter de 120 req/min/IP. Para un admin legitimo es mas que suficiente. Si un Admin se ve comprometido, 120 req/min es alto para enumerar el catalogo.

**Mitigacion.** Aceptable para piloto. Considerar `RequireRateLimiting("public")` (30/min/IP) en Fase 2 si los reports admin son chequeados con frecuencia.

**Estado.** Informational.

## 5. Riesgos aceptados con mitigacion documentada

### F-26 — Race conditions con `xmin` optimistic concurrency

**Riesgo.** Concurrent updates al mismo Order (ej. cocinero marca como "ready" mientras admin marca como "cancelled") pueden colisionar y producir state corrupto.

**Mitigacion temporal.** Volumen del piloto bajo (1 cocinero, 1 admin, ~30-50 orders/dia). Probabilidad de colision real: muy baja. El handler ya hace `db.SaveChangesAsync` que detecta conflicts a nivel SQL en algunos casos (FK violations, etc.). Aceptable para piloto.

**Cierre planeado.** Sesion dedicada post-piloto, agregar `xmin` column como concurrency token EF y test de race con dos contextos.

### F-17 — MFA TOTP para Admin

**Riesgo.** Admin password es la unica defensa contra acceso al panel. Phishing exitoso = control total del chef.

**Mitigacion temporal.** El piloto tiene UN admin (el chef), sus credenciales son personales y no compartidas. Password fuerte (D-04) + JWT TTL de 15 min (F-11) limitan ventana de uso si una sesion se filtra. Refresh tokens rotables con detection de reuso (Pasada A).

**Cierre planeado.** Sesion dedicada con UI work en admin_web (Flutter): formulario de TOTP enrollment + QR code + step en login. Costo estimado 2-3 dias.

## 6. Runbook de deploy

### 6.1 Prerequisites

- VPS Linux con Docker + docker-compose 2.x.
- DNS apuntando `homechef.app` y subdominios al VPS.
- Cuenta Let's Encrypt accesible (no rate-limited por 7 dias previos).

### 6.2 Configurar `deploy/.env`

```bash
cp deploy/.env.example deploy/.env  # si no existe, crearlo a mano
chmod 600 deploy/.env

# Generar secretos
JWT_SIGNING_KEY=$(openssl rand -base64 64)
POSTGRES_PASSWORD=$(openssl rand -base64 24)
REDIS_PASSWORD=$(openssl rand -base64 24)
BOOTSTRAP_ADMIN_PASSWORD=$(openssl rand -base64 18)

# Editar deploy/.env con los valores generados +:
#   AllowedHosts=homechef.app;api.homechef.app;admin.homechef.app
#   Cors__AllowedOrigins__0=https://app.homechef.app
#   Cors__AllowedOrigins__1=https://admin.homechef.app
#   Cors__AllowedOrigins__2=https://kitchen.homechef.app
#   ConnectionStrings__PostgreSQL=Host=postgres;Database=homechef;Username=homechef;Password=${POSTGRES_PASSWORD}
#   ASPNETCORE_ENVIRONMENT=Production
```

### 6.3 Bootstrap de servicios

```bash
docker-compose -f deploy/docker-compose.yml -f deploy/docker-compose.prod.yml up -d postgres redis
sleep 10
docker-compose -f deploy/docker-compose.yml -f deploy/docker-compose.prod.yml up -d api
sleep 5
docker-compose -f deploy/docker-compose.yml -f deploy/docker-compose.prod.yml up -d nginx
```

### 6.4 Aplicar schema de negocio

```bash
docker cp src/database/schema/ homechef-postgres:/tmp/schema/
docker exec -i homechef-postgres bash -c "cd /tmp/schema && psql -U homechef -d homechef -f 99_run_all.sql"
```

### 6.5 Actualizar datos del chef piloto

```bash
docker exec -i homechef-postgres psql -U homechef -d homechef -c "
UPDATE chefs SET
    rif='J-{REAL-RIF}',
    legal_name='{Razon social SENIAT}',
    tax_address='{Direccion fiscal}',
    contact_email='{email del chef}',
    contact_phone='{telefono}'
WHERE id='00000000-0000-0000-0000-000000000001';"
```

### 6.6 Emitir HTTPS cert

```bash
bash deploy/scripts/init-letsencrypt.sh homechef.app api.homechef.app admin.homechef.app
# Activar block HTTPS en nginx/conf.d/api.conf y descomentar redirect HTTP->HTTPS
docker-compose -f deploy/docker-compose.yml -f deploy/docker-compose.prod.yml restart nginx
```

### 6.7 Smoke test post-deploy

```bash
# Health
curl -s https://api.homechef.app/health | jq
# -> {"status":"ok","service":"HomeChefPro.Api"}

# Login del admin
curl -s -X POST https://api.homechef.app/api/auth/login     -H 'Content-Type: application/json'     -d '{"email":"admin@homechef.local","password":"<BOOTSTRAP_ADMIN_PASSWORD>"}' | jq

# Crear menu (vacio al principio, debe responder array vacio)
curl -s https://api.homechef.app/api/client/menu | jq
```

### 6.8 Onboarding del chef piloto

Sesion sincronica con el chef:
1. Cambiar password del admin desde el panel (URL: https://admin.homechef.app).
2. Cargar 5-10 ingredientes basicos.
3. Cargar 3-5 platos del menu.
4. Hacer una orden de prueba E2E (cliente → kitchen → ready → delivered).
5. Verificar receipt PDF y que SENIAT recibe la factura mock.
