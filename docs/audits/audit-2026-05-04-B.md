# Audit Pasada B — Uploads, payment proof, BOLA/BOPLA en orders

**Fecha:** 2026-05-04
**Scope:** `Endpoints/ClientOrdersEndpoints.cs`, `Endpoints/AdminOrdersEndpoints.cs`, `Endpoints/AdminPaymentsEndpoints.cs`, `Endpoints/DeliveryWebhookEndpoints.cs`, `DeliveryWebhookSignatureVerifier.cs`, `Application/Orders/Commands/*`, `Application/Payments/Commands/*`
**Estándar de referencia:** OWASP Top 10 2025, OWASP API Security Top 10 2023 (foco en BOLA/BOPLA/BFLA), OWASP ASVS 5.0 Level 2
**Auditor:** Claude (skill `security-audit`)
**Pasada anterior:** A (auth/JWT/secretos/BFLA admin) — 4 Criticals cerrados, 7 Highs pendientes
**Pasada siguiente:** C (multi-tenant readiness)

---

## 1. Vulnerability summary

```
Critical:  4   (F-22, F-23, F-24, F-32)
High:      5   (F-25, F-26, F-27, F-28, F-31)
Medium:    3   (F-29, F-30, F-33)
Low/Info:  2   (F-34, F-35)
```

**Resumen ejecutivo.** El flujo de orders + payment proof es la zona de mayor riesgo del sistema y donde se concentran los modos #3 (pago manual no escala) y #6 (leak de comprobantes) del premortem. Los hallazgos se agrupan en 3 patrones:

1. **Endpoints anónimos sin tokens de prueba.** Todo `/api/client/orders/*` es `AllowAnonymous` y la única "auth" es conocer el `OrderId` (GUID). El comentario en el código admite el bug textualmente: *"Tracking endpoint — returns order detail by id. In a real rollout this should require a light proof (matching phone / short token). For now it's by id."* (F-24). Combinado con F-22/F-23, cualquiera puede inyectar payment proofs por orders ajenos, leerlos y descargar receipts PDFs con datos personales.

2. **Validación incompleta de campos del body.** `SubmitPaymentProof` acepta `AmountUsd` y `ProofImageUrl` directamente del cliente sin verificar que (a) el monto coincida con `order.TotalUsd`, (b) la URL del comprobante apunte al storage propio. BOPLA puro (API3:2023). F-22 + F-23.

3. **Webhook de delivery acepta requests sin firma cuando el secret está vacío.** En `deploy/.env` los 3 secrets están vacíos por default; el verifier devuelve `null` (no aplica) y el endpoint sigue adelante. F-32 (refinamiento de F-08 de Pasada A).

Adicionalmente, sin rate limiting (F-04 de Pasada A, aún pendiente), los endpoints anónimos de orders/payments son vector de DoS y abuse a escala.

---

## 2. Threat model

**Atacantes considerados.** Anónimo de internet conociendo o adivinando `OrderId`s (GUIDs son 128 bits — no enumerables, pero filtrables vía logs / screenshots / shared links / referrer headers). Atacante con acceso al WhatsApp del chef que reenvía URLs por error. Cliente legítimo que abusa del flujo (auto-aprobar su propio pago). Bot/scraper que crea orders masivos.

**Trust boundaries.** Cliente HTTP ↔ `/api/client/orders/*` (sin auth); cliente ↔ `/api/uploads/payment-proofs` (sin auth para POST); admin/cashier ↔ `/api/admin/orders` y `/api/admin/payments` (auth Cashier+); webhook provider externo ↔ `/api/webhooks/delivery/*` (HMAC, parcial).

**Sensitive assets.** Orders pendientes (incluyen dirección de delivery, teléfono, items + monto), Payments (datos bancarios del comprador, monto, referencia), Receipts PDFs (todos los datos del order + items + total + datos del chef), comprobantes de pago (capturas con números de cuenta), state machine del order (advance ilegítimo = comida gratis).

**Top 5 técnicas MITRE ATT&CK aplicables.**
- **T1190** Exploit Public-Facing Application — endpoints anónimos sin token.
- **T1213** Data from Information Repositories — lectura cross-cliente de orders/receipts.
- **T1565.001** Stored Data Manipulation — submit payment con monto bajo, advance state ilegítimo.
- **T1499** Endpoint DoS — order creation masiva, upload masivo, webhook spoof.
- **T1078** Valid Accounts — webhook spoofing si secret vacío.

### [CRITICAL] F-22 — `POST /api/client/orders/{id}/payment` AllowAnonymous + no valida `AmountUsd` vs `order.TotalUsd`

- **Affected component:** `ClientOrdersEndpoints.cs` líneas 14, 39-60; `SubmitPaymentProofHandler` (no compara `AmountUsd` contra `order.TotalUsd`).
- **Attacker profile:** anónimo de internet con conocimiento de un `OrderId`.
- **Standard:** OWASP API3:2023 BOPLA; OWASP A06:2025 Insecure Design.
- **CWE:** CWE-915 Improperly Controlled Modification of Dynamically-Determined Object Attributes; CWE-840 Business Logic Errors.
- **Description.** El cliente envía `body.AmountUsd` y el handler lo persiste tal cual en `Payment.Submit(...)`. No hay comparación con `order.TotalUsd`. Un atacante puede:
  1. Crear order legítimo de 50 USD (o usar un order ajeno cuyo GUID consiguió).
  2. `POST {id}/payment` con `AmountUsd: 0.01`, comprobante real del propio atacante por 0.01 USD.
  3. Si el admin no nota la discrepancia (especialmente en hora pico), aprueba el payment → order pasa a `paid` → kitchen prepara → comida gratis.
- **Exploitation scenario.**
  1. Crear order: `POST /api/client/orders` con items reales. Recibo `OrderId` y veo `TotalUsd: 50`.
  2. `POST /api/client/orders/{id}/payment` con `{ AmountUsd: 0.01, PaidCurrency: "USD", AmountPaidCurrency: 0.01, ProofImageUrl: "https://homechef.app/api/uploads/payment-proofs/<mio-real-de-1c>.png" }`.
  3. Admin ve "pago de 0.01 USD" pero el comprobante es real → puede aprobar si distrae.
- **Impact:** comida gratis para el atacante con frecuencia inversamente proporcional a la atención del admin. Combinado con DoS (F-28) ataca la economía del chef directamente.
- **Recommended fix.**
  - En `SubmitPaymentProofHandler`, agregar: `if (request.AmountUsd != order.TotalUsd) throw new ConflictException("Payment amount must match order total.");`. Rejection con 409.
  - Idealmente, NO aceptar `AmountUsd` del body. Derivarlo server-side de `order.TotalUsd`. El cliente solo declara `PaidCurrency`, `AmountPaidCurrency` (en VES si paga en bolívares), `ExchangeRateUsed`, y `ReferenceNumber`.
  - Validación cruzada: `AmountPaidCurrency / ExchangeRateUsed == order.TotalUsd ± tolerance` cuando `PaidCurrency=VES`.
- **Effort to fix:** small.
- **Verification:** test integration `Submit_payment_with_wrong_amount_should_be_rejected_409` que crea order de 50 USD y submite con `AmountUsd: 0.01` → 409.

### [CRITICAL] F-23 — `ProofImageUrl` aceptada sin validación que apunte al storage propio

- **Affected component:** `ClientOrdersEndpoints.cs` línea 53; `SubmitPaymentProofCommand` línea 20; `SubmitPaymentProofValidator` (sin regla para `ProofImageUrl`).
- **Attacker profile:** anónimo, cliente abusivo.
- **Standard:** OWASP API3:2023 BOPLA; OWASP A10:2021 SSRF (consolidado en A01:2025); OWASP ASVS V5.5 (Deserialization & redirects).
- **CWE:** CWE-601 Open Redirect; CWE-918 SSRF; CWE-829 Inclusion of Functionality from Untrusted Control Sphere.
- **Description.** El campo `ProofImageUrl` es `string?` sin validación de formato ni de host. El atacante puede pasar:
  - URL externa controlada por él (`https://attacker.com/legit-looking.png`) — el admin web la abre para ver el comprobante. Si renderiza inline puede ser tracking pixel; si abre en pestaña nueva puede ser landing de phishing.
  - URL con esquema `javascript:` o `data:` — si el admin web la usa en `<img src>` o en `<a href>` sin sanitizar, ejecuta JS en el contexto del admin (XSS / cookie theft).
  - URL del propio storage **apuntando a comprobante de OTRO cliente** (`https://homechef.app/api/uploads/payment-proofs/<guid-de-otro>.png`) — re-uso de comprobante real que el admin ya verificó. El admin "ya conoce ese comprobante" y aprueba sin re-verificar.
  - URL interna (`http://localhost:5432/`) — SSRF si el server-side hace fetch para preview/thumbnail.
- **Exploitation scenario** (re-uso de comprobante).
  1. Atacante envía `POST /payment` legítimo con `AmountUsd: 50` y comprobante real propio. Admin aprueba.
  2. Atacante crea otro order y obtiene la URL del comprobante anterior (de su propia respuesta API).
  3. Submit payment del nuevo order pasando la **misma** `ProofImageUrl`. Admin ve un comprobante "que reconoce" y aprueba.
  4. Mismo comprobante usado para N orders.
- **Impact:** comida gratis a escala, phishing del admin, posible XSS, posible SSRF.
- **Recommended fix.**
  - Cambiar el flujo: en lugar de `ProofImageUrl: string`, recibir `ProofImageId: Guid` que sea el ID retornado por el `POST /api/uploads/payment-proofs`. El handler hace lookup en una tabla de uploads pendientes y verifica que (a) existe, (b) no está asociado todavía a otro payment, (c) fue subido recientemente (timeout de 15 min).
  - Si por compatibilidad debe seguir siendo URL: regex que solo acepte `^/api/uploads/payment-proofs/[a-f0-9]{32}\.(jpg|jpeg|png|webp)$` (relativo, sin host externo). Y verificar que ese filename no esté ya asociado a otro payment (`UNIQUE` constraint en BD).
- **Effort to fix:** medium.
- **Verification:** tests que rechacen URLs externas, `javascript:`, URLs reusadas, URLs malformadas.

### [CRITICAL] F-24 — `GET /api/client/orders/{id}` y `/receipt.pdf` AllowAnonymous + sin token de ownership

- **Affected component:** `ClientOrdersEndpoints.cs` líneas 27-37. El comentario línea 27 dice textualmente: *"Tracking endpoint — returns order detail by id. In a real rollout this should require a light proof (matching phone / short token). For now it's by id."*
- **Attacker profile:** anónimo con un `OrderId` filtrado.
- **Standard:** OWASP API1:2023 BOLA; OWASP A01:2025 BAC; ASVS V4.2.1.
- **CWE:** CWE-639 Authorization Bypass Through User-Controlled Key.
- **Description.** Cualquiera con un `OrderId` (GUID) puede leer todos los detalles del order: items, montos, dirección de delivery, teléfono del cliente, notas. Y descargar el receipt PDF que típicamente incluye RIF del chef, datos del cliente, items, total con IGTF/IVA. Los GUIDs son 128 bits (no enumerables), pero se filtran fácilmente:
  - Logs centralizados / Serilog en stdout que capturan request URLs.
  - Screenshots de WhatsApp del cliente reenvíadas.
  - URLs en Slack, email, browser history.
  - Referrer header si el cliente clickea un link externo desde la página del order.
  - SQLi / XSS en otra parte del sistema que filtre `order_id`s.
- **Exploitation scenario.**
  1. Atacante consigue 1 `OrderId` (vía cualquier vía de filtración).
  2. `GET /api/client/orders/{id}` → datos personales del cliente (nombre, teléfono, dirección).
  3. `GET /api/client/orders/{id}/receipt.pdf` → PDF con datos completos.
  4. Si por iteración temporal (`created_at` cercano) consigue otros `order_id`s, escala a leak masivo.
- **Impact:** exposición de PII (dirección, teléfono, nombre completo) y datos financieros (montos, items, RIF). Mismo blast radius que F-02 de Pasada A.
- **Recommended fix.**
  - Agregar `accessToken` GUID corto al order al crearse (aparte del `id`). El cliente recibe `{ id, accessToken }` en la respuesta de POST. Para `GET` debe enviar `?token=<accessToken>` o header `X-Order-Token`. Server-side compara con `accessToken` almacenado.
  - O exigir `?phone=<last4>` o similar como token ligero. El comentario del código mismo lo sugiere.
  - O requerir auth para clientes registrados; para guests, accessToken corto.
- **Effort to fix:** medium.
- **Verification:** test `Get_order_without_token_should_fail` y `Get_order_with_wrong_token_should_fail`.

### [CRITICAL] F-32 — Webhook delivery acepta requests sin firma cuando secret está vacío en config

- **Affected component:** `DeliveryWebhookSignatureVerifier.cs` líneas 22-26 y 38-40; `DeliveryWebhookEndpoints.cs` línea 48 (`if (sigValid == false)` — solo rechaza explícitamente `false`, no `null`); `deploy/.env` (3 secrets vacíos por default).
- **Attacker profile:** anónimo de internet.
- **Standard:** OWASP A08:2025 Software and Data Integrity Failures; OWASP API10:2023 Unsafe Consumption of APIs.
- **CWE:** CWE-345 Insufficient Verification of Data Authenticity; CWE-1392 Use of Default Credentials (semánticamente).
- **Description.** El verifier construye `_secrets` solo con providers que tengan secret no-vacío. Si el provider no aparece en `_secrets`, `Verify` retorna `null`. El endpoint solo rechaza cuando `sigValid == false` (firma presente y mala). `null` (no hay secret configurado) **deja pasar el evento como "no verificado"**. En `deploy/.env`:
  ```
  DELIVERY_SECRET_YUMMY=
  DELIVERY_SECRET_PEDIDOSYA=
  DELIVERY_SECRET_RAPPI=
  ```
  Todos vacíos. Si jacques despliega sin poblar estos, **cualquiera con la URL** del webhook (que es `/api/webhooks/delivery/{provider}`, anonymous) puede inyectar eventos:
  - `POST /api/webhooks/delivery/yummy` con body `{ "status":"delivered", "orderId":"..." }` → `IngestDeliveryEvent` se ejecuta → `order.MarkDelivered(...)` se aplica.
- **Exploitation scenario.**
  1. Atacante crea legítimamente un order. Recibe `OrderId`.
  2. `POST /api/webhooks/delivery/yummy` con `{ "status":"delivered", "orderId":"<el-mio>" }` sin header de signature.
  3. Verifier retorna `null` (no hay secret en `.env`). Endpoint sigue adelante. `IngestDeliveryEvent` cambia el state del order a `delivered`.
  4. Atacante nunca pagó realmente. Comida ya entregada (kitchen workflow), ahora aparece como completada.
- **Impact:** workflow bypass total del state machine de delivery; fraude.
- **Recommended fix.**
  - **En `Program.cs` startup:** validar que si `DeliveryWebhooks:RejectInvalidSignature == true` (default true), TODOS los providers configurados en `DeliveryWebhooks:Secrets` tengan valor no-vacío. Si alguno está vacío, fail-fast al boot.
  - **En `DeliveryWebhookEndpoints`:** rechazar también `sigValid == null` cuando `RejectInvalidSignature=true`. Solo aceptar `null` cuando explícitamente se permite (modo dev sin secret).
  - **En `Verify`:** invertir lógica. Si el secret está vacío, retornar `false` (no `null`).
  - Documentar en `docs/RUNBOOK.md`: secrets son obligatorios en prod.
- **Effort to fix:** small.
- **Verification:** test `Webhook_without_signature_should_be_rejected_when_secret_empty` y `Webhook_startup_should_fail_if_any_secret_empty_in_prod_mode`.

### [HIGH] F-25 — `SubmitPaymentProofHandler` no rechaza si order ya pasó `PendingPayment`

- **Affected component:** `SubmitPaymentProofHandler` líneas 77-79: `if (order.Status == OrderStatus.PendingPayment) order.SubmitPayment(clock);`. No hay rechazo cuando el order ya está en otro estado.
- **Attacker profile:** cliente abusivo, anónimo con OrderId.
- **Standard:** OWASP A06:2025 Insecure Design; OWASP API6:2023 Unrestricted Access to Sensitive Business Flows.
- **CWE:** CWE-840 Business Logic Errors.
- **Description.** Si un order ya está en `Paid`, `InPreparation`, `Ready`, `InDelivery` o `Delivered`, el handler **igual crea un Payment record nuevo** (sin tirar excepción). Solo no avanza el state. Resultado:
  - Múltiples payments en la BD para el mismo order (duplicate records).
  - Si el admin tiene UI que lista pending payments por `verified_at IS NULL`, los duplicados aparecen aún cuando el order ya está pagado → confusión.
  - Posible re-aprobación si el admin hace click en el segundo payment sin notar que el primero ya está aprobado y el order ya avanzó.
- **Recommended fix.**
  - En el handler: `if (order.Status != OrderStatus.PendingPayment) throw new ConflictException($"Order is in state {order.Status}; cannot accept new payment proof.");`
  - O: state guard a nivel `Payment.Submit(...)` que tire si ya existe Payment unverified para el order.
- **Effort to fix:** small.
- **Verification:** test `Submit_second_payment_when_order_is_paid_should_be_rejected_409`.

### [HIGH] F-26 — Race condition en `SubmitPaymentProof` concurrente sobre el mismo order

- **Affected component:** `SubmitPaymentProofHandler` (sin lock optimista, sin unique constraint en `payments(order_id) WHERE verified_at IS NULL`).
- **Attacker profile:** atacante con scripting básico (curl en bucle).
- **Standard:** OWASP A06:2025 Insecure Design (TOCTOU).
- **CWE:** CWE-362 Race Condition; CWE-367 TOCTOU.
- **Description.** El handler hace `db.Orders.FirstOrDefaultAsync(o => o.Id == ...)`, evalúa `if (order.Status == OrderStatus.PendingPayment) order.SubmitPayment(clock)`, y `db.SaveChangesAsync()`. Sin lock, dos requests concurrentes para el mismo order pueden ambos: (a) leer Status=PendingPayment, (b) crear Payment, (c) llamar `SubmitPayment` (que muta Status a `PaymentVerifying`). EF Core con default optimistic concurrency probablemente no detecte esto si la entidad Order no tiene `RowVersion`/`xmin`. Resultado: 2 payments para 1 order.
- **Recommended fix.**
  - Agregar `xmin` (PostgreSQL system column) como concurrency token en `OrderConfiguration`: `builder.UseXminAsConcurrencyToken();`.
  - Y/O agregar índice unique parcial en BD: `CREATE UNIQUE INDEX ux_payments_one_pending_per_order ON payments(order_id) WHERE verified_at IS NULL AND rejected_at IS NULL;`.
  - Tests con `Parallel.For` 50 requests al mismo endpoint con mismo `OrderId`.
- **Effort to fix:** small (xmin) a medium (con tests).
- **Verification:** test concurrente que dispara 50 requests paralelos → solo 1 Payment record creado.

### [HIGH] F-27 — `AmountUsd / AmountPaidCurrency / ExchangeRateUsed` no se validan ratios coherentes

- **Affected component:** `SubmitPaymentProofValidator` (líneas 38-42 solo valida que `ExchangeRateUsed > 0` cuando VES; no compara con `AmountPaidCurrency / AmountUsd`).
- **Standard:** OWASP API3:2023 BOPLA; A06:2025 Insecure Design.
- **CWE:** CWE-840 Business Logic Errors.
- **Description.** Cliente puede enviar combinaciones absurdas:
  - `AmountUsd: 0.01, AmountPaidCurrency: 1000000, ExchangeRateUsed: 1` (hace que parezca pago grande en VES pero registrado como 0.01 USD).
  - `AmountUsd: 50, AmountPaidCurrency: 1, ExchangeRateUsed: 50` (pago de 1 VES = 50 USD: claramente irreal con tasa actual ~50 VES/USD).
  - El admin debe verificar cada caso manualmente. Sin enforcement automatizado, errores pasan.
- **Recommended fix.**
  - Validar: `Math.Abs((AmountPaidCurrency / ExchangeRateUsed) - AmountUsd) <= 0.01m` cuando `PaidCurrency=VES`.
  - Validar `AmountUsd == AmountPaidCurrency` cuando `PaidCurrency=USD`.
  - Validar tasa VES/USD razonable (ej. entre 10 y 1000 VES/USD; rangos a coordinar con el chef).
  - Idealmente, derivar tasa server-side del último `exchange_rate` registrado en lugar de aceptarla del cliente.
- **Effort to fix:** small.
- **Verification:** tests con combinaciones inválidas → 400.

### [HIGH] F-28 — Endpoints anónimos de orders/payments sin rate limiting (extiende F-04)

- **Affected component:** `ClientOrdersEndpoints` (todo el grupo `AllowAnonymous`); `UploadEndpoints` (POST anonymous).
- **Standard:** OWASP API4:2023 Unrestricted Resource Consumption; OWASP A06:2025.
- **CWE:** CWE-770; CWE-799 Improper Control of Interaction Frequency.
- **Description.** F-04 de Pasada A documentó la ausencia general de rate limiting. En el contexto de orders/payments el riesgo es específico:
  - **Order spam:** anonymous puede crear infinitos orders, llenando la tabla `orders` y `guest_customers`. Cada uno consume queries/inserts.
  - **Payment proof spam:** anonymous puede submitir N payments para un mismo order ajeno (combinado con F-25 que no rechaza). Llena tabla `payments`.
  - **Order tracking spam:** F-24 + GET masivo de receipts PDFs (cada uno genera PDF con QuestPDF, costoso).
  - **Webhook spam:** combinado con F-32, atacante puede inundar `IngestDeliveryEvent`.
- **Recommended fix.**
  - Aplicar rate limit policy `"public"` (10/min/IP, 100/hora/IP) a todos los endpoints AllowAnonymous de orders, payments y webhooks.
  - Y rate limit más estricto a operaciones de costo alto (receipt PDF: 5/min/IP).
- **Effort to fix:** small (cuando F-04 se cierre, agregar `.RequireRateLimiting("public")` a estos endpoints).
- **Verification:** test que dispara 11 GET de receipt.pdf → 11vo y 12vo 429.

### [HIGH] F-31 — `OrderLineInput.Quantity` sin máximo permite orders con totales astronómicos

- **Affected component:** `CreateGuestOrderValidator` línea 43: `RuleFor(i => i.Quantity).GreaterThan(0)`. Sin `LessThanOrEqualTo` máximo.
- **Standard:** OWASP A06:2025 Insecure Design; OWASP API4:2023 Unrestricted Resource Consumption.
- **CWE:** CWE-770; CWE-1284 Improper Validation of Specified Quantity in Input.
- **Description.** Cliente puede crear order con `Quantity: 1000000`. `dish.SellingPriceUsd * 1000000 = N millones USD`. Posibles consecuencias:
  - Si `decimal` overflow (raro en .NET decimal por su rango, pero posible con multiplicaciones encadenadas), excepción + posible state inconsistente.
  - Reportes (margen, sales-daily) muestran números absurdos que rompen dashboards y triggers.
  - Storage attack: cada order con N items grandes consume escritura.
  - Si por error el chef o admin avanza el state sin notar, kitchen recibe orden de preparar 1M de un platillo.
- **Recommended fix.**
  - Agregar a `OrderLineInput`: `RuleFor(i => i.Quantity).GreaterThan(0).LessThanOrEqualTo(50)` (límite razonable para "comida casera"; coordinar con el chef).
  - Y: validar `Items.Count <= 30` (prevenir orders con 1000 líneas).
  - Y: validar `order.TotalUsd <= 5000` o similar (cap absoluto ajustable).
- **Effort to fix:** trivial.
- **Verification:** test con `Quantity: 1000000` → 400.

### [MEDIUM] F-29 — Stock no se decrementa al crear order (overselling)

- **Affected component:** `CreateGuestOrderHandler` (no toca `Recipe.CurrentStock` ni `IngredientStock`).
- **Standard:** OWASP A06:2025 Insecure Design.
- **CWE:** CWE-840 Business Logic Errors; CWE-362 (cuando es concurrente).
- **Description.** Cuando se crea un order, el stock disponible no se decrementa atómicamente. Si solo hay 1 unidad de `tequeños` y 100 personas crean order al mismo tiempo, todos los 100 orders se crean exitosamente. Solo se valida `IsOutOfStock` AL MOMENTO DE LA CREACIÓN (que probablemente lee el stock cacheado, no decrementa).
- **Recommended fix.**
  - Agregar tabla `inventory_reservations(order_id, dish_id, quantity, created_at, expires_at)`.
  - Al crear order, INSERT con `expires_at = now + 15 min`. Al pagar, convertir reservation en consumption.
  - Trigger SQL que valida `SUM(reservations.quantity) <= dish.current_stock` antes de cada INSERT con `LOCK TABLE` o advisory lock.
  - Cleanup cron que libera reservas expiradas.
- **Effort to fix:** medium-large (cambio de modelo de inventario).
- **Verification:** test concurrente `Should_reject_orders_above_stock`.

### [MEDIUM] F-30 — `GuestCustomer` se crea sin throttle por cada order anonymous

- **Affected component:** `CreateGuestOrderHandler` línea 82-83: `var guest = GuestCustomer.Create(...); db.GuestCustomers.Add(guest);`.
- **Standard:** OWASP API4:2023; A06:2025.
- **CWE:** CWE-770.
- **Description.** Sin rate limiting (F-04, F-28), cada call a `POST /api/client/orders` crea un nuevo `GuestCustomer`. Atacante puede llenar la tabla con millones de registros sintéticos. Aún sin order completo (si la creación falla por validación de items, ya puede ya haberse insertado el guest si hay timing entre `Add` y `SaveChanges`).
- **Recommended fix.**
  - Reusar `GuestCustomer` por `(GuestPhone, GuestFullName)` si ya existe en último N días.
  - Combinar con F-28 (rate limit en order creation).
- **Effort to fix:** small.

### [MEDIUM] F-33 — `PayerAccountLast4.MaximumLength(10)` permite más de 4 caracteres

- **Affected component:** `SubmitPaymentProofValidator` línea 46: `RuleFor(x => x.PayerAccountLast4).MaximumLength(10)`.
- **Standard:** ASVS V5.1 Input Validation.
- **CWE:** CWE-20 Improper Input Validation.
- **Description.** El campo se llama "last 4" pero acepta hasta 10 chars. Esto es un input validation issue menor pero genera ambigüedad: ¿son los últimos 4 dígitos? ¿Los últimos 10? Si el admin ve un valor de 8 chars, ¿cómo interpreta? Puede ser explotable si el frontend del admin no sanitiza al mostrar.
- **Recommended fix.**
  - Cambiar a `RuleFor(x => x.PayerAccountLast4).Matches(@"^\d{4}$")` con mensaje "Exactly 4 digits required."
  - O renombrar el campo a algo más descriptivo si el intent era distinto.
- **Effort to fix:** trivial.

### [LOW] F-34 — `AdvanceOrderStatus` permite cualquier transición pasando por el FSM del domain

- **Affected component:** `AdvanceOrderStatusHandler`. Confiamos en que `Order.SubmitPayment/ApprovePayment/StartPreparation/...` validen las transiciones. **Pendiente de auditar el dominio Order**.
- **Standard:** A06:2025 Insecure Design.
- **CWE:** CWE-840.
- **Description.** El handler asume que cada método del dominio tira si la transición no es válida. ## 4. Attack chains

### CH-B1 — "Comida gratis" — F-22 + F-23 + F-25 + F-28

**Goal:** ordenar comida real sin pagar.

**Pre-conditions:** F-22 (no valida amount) + F-23 (URL forjable) + F-25 (no rechaza re-submit).

**Steps:**
1. Atacante hace `POST /api/client/orders` con items reales. Recibe `OrderId`.
2. `POST /api/client/orders/{id}/payment` con `AmountUsd: 0.01` y un `ProofImageUrl` apuntando a un comprobante real propio de 0.01 USD (que sí pagó).
3. Si admin no nota, aprueba. Order → Paid → Kitchen prepara.
4. Si admin rechaza, atacante hace re-submit con `AmountUsd: 50` real esta vez. Pero por F-25 puede submitir múltiples — confunde el queue del admin.
5. Sin rate limit (F-28), repetir N veces.

**MITRE ATT&CK:** T1565.001 → T1499.

**Mitigation:** F-22 + F-23 + F-25 + F-28 fix juntos.

## 5. Secure design recommendations

### Architectural

1. **Token de ownership ligero para orders anonymous.** Generar `accessToken` GUID corto al crear order. Toda operación `/api/client/orders/{id}/*` requiere `?token=` o header.
2. **`ProofImageUrl` debe ser `ProofImageId`** (referencia al upload server-side, no URL libre). Tabla `pending_uploads(id, filename, uploaded_at, claimed_at)` con TTL.
3. **State machine guards en BD, no solo en domain.** Constraints check + triggers que rechacen transiciones inválidas a nivel SQL. Defense in depth.
4. **Webhook fail-closed por default.** Si secret vacío en config, `null` debe ser tratado como `false` (rechazar).
5. **Inventory reservations** para resolver overselling (F-29) antes del primer cliente real.

### Patrones

- Nunca aceptar `AmountUsd` del body cuando ya se conoce `order.TotalUsd`. Derivar server-side.
- Validar URLs estrictamente con regex anclada al storage propio.
- `xmin` como concurrency token en todas las entidades con state machine (Order, Payment, Invoice).
- Log estructurado con `correlation_id` + `order_id` para trazabilidad. NO loguear datos bancarios en plain text.

### ASVS gaps detectados (Pasada B)

- V4.2.1 (Object-level authorization en GET orders) — gap.
- V5.1 (Input validation completa en payment proof) — gap parcial.
- V8.3 (Sensitive data not exposed via insecure direct object reference) — gap.
- V11.1 (Business logic flaws — workflow bypass) — gap.
- V13.1 (API security — anonymous bound endpoints sin rate limit) — gap (heredado de F-04).
